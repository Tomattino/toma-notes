インデックス 教材

仕組みから実務アンチパターンまで — プロに必須の知識

計算量とは — O記法の基礎知識

インデックスの説明でよく出てくる O(n) や O(log n) は「データが増えたときに処理時間がどう増えるか」を表す記法（O記法・ビッグオー記法）。

記法意味具体例100万行のとき
O(1)データ量に関係なく一定主キーで1件取得1回
O(log n)データが倍になっても処理は1回増えるだけB-treeインデックス検索約20回
O(n)データに比例して増えるフルスキャン100万回
O(n²)データの二乗で増えるインデックスなしのJOIN1兆回（実用不可）

インデックスを使うと O(n)（フルスキャン）から O(log n)（B-tree検索）になる。100万行のテーブルで比べると「100万回 vs 約20回」の差。

なおO記法はあくまで「データが増えたときの増え方のざっくりした傾向」を表すもので、実際の回数を正確に計算する式ではない。B-treeの内部で1ノードに何個キーを持つか（分岐数）はDBが自動で最適化するもので、こちらから指定することはできないし気にする必要もない。「インデックスがあれば劇的に速くなる」という傾向を示す表現と理解しておけば十分。

それぞれのイメージ

記法日常のたとえなぜその速さか
O(1) 引き出しの中の特定の仕切りを直接開ける 主キーで検索するとDBは「この行はここにある」と直接アドレスを知っている。何行あっても1回
O(log n) 辞書で単語を探すとき、真ん中を開いて「前か後か」を繰り返す 100万件あっても「前半か後半か」を繰り返すだけで約20回で到達。倍になっても+1回しか増えない
O(n) 部屋の中に落としたイヤホンを端から端まで探す インデックスがないと先頭から1行ずつ全部見るしかない。100万行あれば100万回チェック
O(n²) クラス全員が他の全員と1回ずつ握手する（30人なら870回） インデックスなしのJOINは「テーブルAの各行に対してテーブルBを全件スキャン」する。1万行 × 1万行 = 1億回

インデックスとは何か — B-tree構造の仕組み

インデックスは「本の索引」と同じ仕組み。本文を先頭から読まなくても、索引を引けば目的のページにすぐ飛べる。

DBのインデックスは B-tree（バランス木） という構造でデータを管理している。

フルスキャンB-treeインデックス
計算量O(n) — 行数に比例して遅くなるO(log n) — 行数が増えても対数的にしか増えない
100万行で検索最大100万回の比較最大約20回の比較
1億行で検索最大1億回最大約27回
インデックスの効果は行数が増えるほど劇的に大きくなる。100行のテーブルでは差がわからなくても、100万行になると桁違いの差が出る。

SARGability — インデックスを活かす書き方・殺す書き方

SARGable（サージャブル） = Search ARGument ABLE。インデックスを使った検索が可能かどうかを表す。インデックスを貼っていても、書き方次第でフルスキャンになってしまう。

なぜ関数をかけるとインデックスが使えなくなるのか

B-treeインデックスは「そのカラムの値そのもの」を索引として管理している。

WHERE YEAR(created_at) = 2024 と書くと、DBは「YEAR(created_at) の結果」で検索しようとする。でもインデックスには created_at の値（例: 2024-03-15 10:00:00）しか入っていない。YEAR(created_at) の結果（2024）はどこにも索引として存在しない。

結果として、全行に対して YEAR(created_at) を実行して計算してから比較するしかなくなる = フルスキャン。

ルール: インデックスを使うには、左辺のカラムを「そのまま」比較する必要がある。関数・演算・型変換をかけると死ぬ。

カラムに関数・演算を適用するとインデックスが使えない

✗ NG — インデックス無効（フルスキャン）
-- カラムに関数を適用している
SELECT * FROM users
WHERE YEAR(created_at) = 2024;

-- カラムを演算している
SELECT * FROM orders
WHERE amount * 1.1 > 10000;

-- 暗黙の型変換が起きている
SELECT * FROM users
WHERE user_code = 12345;
-- ↑ user_code が VARCHAR型なのに数値で比較
✓ OK — インデックス有効
-- 範囲指定に書き換える
SELECT * FROM users
WHERE created_at >= '2024-01-01'
  AND created_at <  '2025-01-01';

-- 右辺に演算を移す
SELECT * FROM orders
WHERE amount > 9091;
-- ↑ 10000/1.1 を先に計算して定数にする

-- 型を合わせる
SELECT * FROM users
WHERE user_code = '12345';
-- ↑ 文字列で比較

LIKE のワイルドカードの位置

✗ NG — 前方ワイルドカードはインデックス無効
-- 先頭に % があるとインデックス使えない
SELECT * FROM users
WHERE name LIKE '%山田%';

SELECT * FROM users
WHERE name LIKE '%田';

B-treeは「前から何文字目か」で索引を引く。先頭が不定だと引けない。

✓ OK — 後方ワイルドカードはインデックス有効
-- 末尾だけなら使える
SELECT * FROM users
WHERE name LIKE '山田%';
-- ↑「山田」から始まるものをB-treeで絞れる

前方一致なら索引を「山田」から引いていける。

「名前に◯◯を含む」のような中間一致検索が必要なら、MySQLならFULLTEXT INDEX、PostgreSQLならGINインデックス＋pg_trgm拡張を使う。通常のB-treeインデックスでは対応できない。

OR条件とインデックスの関係

OR条件はインデックスの扱いが複雑になる。基本的な考え方は「OR のどちらか一方でもインデックスが使えない条件があると、全体がフルスキャンになりうる」。

ただし最近のMySQL・PostgreSQLには Index Merge という最適化があり、両側のカラムにインデックスがあれば OR でも各インデックスを個別に使って結果をマージすることがある。あくまでDBの判断次第で保証はされない。

対処の優先順位：

① まず両カラムにインデックスを追加する（最もシンプルで確実）
② インデックスを追加できない事情がある場合のみ、UNION に書き換えることを検討する

UNIONは同じテーブルに2回クエリを発行することになるが、フルスキャン1回よりインデックス検索2回の方が速い。これはO(n)とO(log n)の差が数万倍以上になることがあるため。

-- NG: OR の一方がインデックスなしカラムだと全体がフルスキャンになることがある
SELECT * FROM users
WHERE email = '[email protected]'  -- インデックスあり
   OR  name  = '山田';           -- インデックスなし → 全体がフルスキャンになりうる

-- 最善: name にもインデックスを追加する
CREATE INDEX idx_users_name ON users (name);
-- これで OR 条件でも Index Merge が使われる可能性が高まる

-- 回避策: UNION で分割するとそれぞれのインデックスを確実に使える
-- 2回実行するがフルスキャン×1回よりインデックス×2回の方が速い
SELECT * FROM users WHERE email = '[email protected]'
UNION
SELECT * FROM users WHERE name  = '山田';

フルスキャンの恐怖 — 無計画なJOINとLIKEがサーバーを殺す

JOINアルゴリズムの種類（DBが自動選択）

DBはJOINの際に内部で結合アルゴリズムを自動選択する。インデックスの有無で使われるアルゴリズムが変わる。

アルゴリズム仕組み使われる条件
Nested Loop Join外側テーブルの各行に対して内側テーブルをループ検索する。インデックスがあれば内側の検索がO(log n)になるので高速。小〜中規模、インデックスがある場合
Hash Join片方のテーブルをハッシュテーブルに変換してからもう片方と照合する。大量データでも安定して速いが、メモリを消費する。大規模、インデックスがない場合
Merge Join両テーブルをソート済みにしてから順番に結合する。両側にインデックスがあれば効果的。ソート済みデータ、インデックスがある場合

基本はDBオプティマイザが自動選択するので意識する必要は薄いが、EXPLAINで確認できる。インデックスなしのNested Loop Joinが最も危険（O(n²)になる）。

JOINはON句のカラムにインデックスが必要

JOINのON句で指定するカラムにインデックスがないと、結合のたびに相手テーブルを全件スキャンする。

-- JOINのON句カラムには必ずインデックスをつける
CREATE INDEX idx_orders_user_id ON orders (user_id);

-- インデックスがあればHash JoinまたはIndex Nested Loop Joinが使われる
SELECT
    u.name,
    COUNT(o.id) AS order_count
FROM       users  u
INNER JOIN orders o ON u.id = o.user_id  -- user_idにインデックスがあれば高速
GROUP BY   u.id, u.name;

SELECT * の弊害

必要なカラムだけ指定することで、インデックスだけで完結する（後述のカバリングインデックス）ケースが増える。

-- NG: 全カラム取得は転送量も多くインデックスの恩恵を受けにくい
SELECT * FROM users WHERE email = '[email protected]';

-- OK: 必要なカラムだけ
SELECT id, name FROM users WHERE email = '[email protected]';

複合インデックス — カラムの順番が命

複数カラムを組み合わせたインデックス。左端のカラムから順にしか使えないというルールがある。

-- 例: department_id と status の複合インデックス
CREATE INDEX idx_users_dept_status ON users (department_id, status);
WHERE句の条件インデックスの使われ方判定
WHERE department_id = 1department_id 部分を使う✓ 有効
WHERE department_id = 1 AND status = 'active'両カラムを使う（最も効果的）✓ 有効
WHERE status = 'active'使えない（左端のdepartment_idをスキップできない）✗ 無効
WHERE status = 'active' AND department_id = 1書く順番は関係ない。両カラムがあればOK✓ 有効
⚠ 左端カラムのルール（プレフィックスルール） (A, B, C) の複合インデックスは以下のように使われる。
A のみ → 使える / A + B → 使える / A + B + C → 使える
B のみ → 使えない / C のみ → 使えない / B + C → 使えない

カーディナリティ（値の種類の多さ）が高いカラムを左に置くのが基本。

複合インデックスの設計指針

指針理由
等値条件（=）のカラムを先に範囲条件（>, <, BETWEEN）の後ろにあるカラムは使われない
選択性（カーディナリティ）が高いカラムを先に値の種類が多いほど絞り込み効果が高い
ORDER BY のカラムを末尾に含めるソート処理をインデックスで代替できる
-- 例: 等値条件のdepartment_idを先に、範囲条件のcreated_atを後に
CREATE INDEX idx_users_dept_created ON users (department_id, created_at);

-- このクエリに最適
SELECT * FROM users
WHERE    department_id = 1              -- 等値条件（先のカラム）
  AND  created_at >= '2024-01-01';   -- 範囲条件（後のカラム）

カバリングインデックス — テーブルアクセスをゼロにする

クエリで必要なカラムが全てインデックスに含まれている場合、テーブル本体へのアクセスが不要になる。これをカバリングインデックス（Index Only Scan）という。

-- カバリングインデックスのカラム順のルール:
-- 先頭: WHERE句・JOINで使う検索条件カラム（最重要。ここでカーディナリティを考慮）
-- 末尾: SELECTで取得したいがWHEREには使わないカラム（おまけ扱いでOK）
CREATE INDEX idx_users_email_covering ON users (email, id, name);
--                                         ↑先頭  ↑末尾（カバリング目的で追加）
-- id はカーディナリティが高いが検索条件に使わないので末尾でOK

-- このクエリはテーブル本体にアクセスしない（インデックスだけで完結）
SELECT id, name          -- インデックスに含まれているカラムだけ
FROM   users
WHERE  email = '[email protected]';

-- NG: * で取得するとインデックスにないカラムが必要になりテーブルアクセスが発生する
SELECT *
FROM   users
WHERE  email = '[email protected]';
カバリングインデックスは特に読み取りが多いAPIエンドポイントで効果を発揮する。ただしインデックスのサイズが大きくなるのでトレードオフを考慮する。

EXPLAIN — 実際に何が起きているか確認する

インデックスが効いているかどうかは EXPLAIN（実行計画）で確認できる。コードレビューやパフォーマンス改善の際に必ず使う。

-- MySQL: EXPLAINをSELECTの前につけるだけ
EXPLAIN SELECT * FROM users WHERE email = '[email protected]';

-- PostgreSQL: EXPLAIN ANALYZEで実際の実行時間も表示
EXPLAIN ANALYZE SELECT * FROM users WHERE email = '[email protected]';

インデックスが逆効果になるケース

ケース理由対処
書き込みが極めて多いテーブルINSERT/UPDATE/DELETE のたびにインデックスも更新コストがかかるインデックス数を絞る。バッチ処理中は一時的に無効化も検討
カーディナリティが低いカラム男女（2値）や bool カラムはインデックスで絞り込んでもほぼ半分しか絞れずフルスキャンと変わらないインデックスをつけない。複合インデックスの末尾に入れるのはOK
小さいテーブル（数百行程度）フルスキャンの方がオーバーヘッドが少ない。DBオプティマイザが自動でフルスキャンを選ぶインデックスをつけても無駄になる。気にしなくていい
インデックスを大量につけすぎる書き込み性能の低下、メモリ・ディスク消費増加実際のクエリパターンに絞って最小限に

実務でよくやるアンチパターン集

AP-1 WHERE句でカラムに関数を使う

✗ NG

WHERE DATE(created_at) = '2024-01-01'
WHERE UPPER(name) = 'YAMADA'
WHERE SUBSTRING(code, 1, 3) = 'ABC'

✓ OK

WHERE created_at >= '2024-01-01'
  AND created_at <  '2024-01-02'
WHERE name = 'yamada' -- 正規化して格納
WHERE code LIKE 'ABC%'
AP-2 N+1問題 — ループ内でSQLを発行する

アプリコード側の問題だが、結果的にDBへの大量クエリになる。

✗ NG（N+1）

-- まずユーザー一覧を取得（1回）
users = SELECT * FROM users;

-- その後ユーザーごとに注文を取得（N回）
for user in users:
    orders = SELECT * FROM orders
             WHERE user_id = user.id;

✓ OK（JOINで1回にまとめる）

SELECT
    u.id, u.name,
    o.id AS order_id, o.amount
FROM       users  u
LEFT JOIN  orders o
    ON u.id = o.user_id;
AP-3 インデックスのないカラムでORDER BY / GROUP BY

ソートや集計のために一時テーブル（filesort）が作られ、大量メモリを消費する。EXPLAINの Extra に Using filesort や Using temporary が出たら要注意。

-- EXPLAINで確認
EXPLAIN SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20;
-- Extra: Using filesort ← インデックスなしでソートしている

-- 対処: ORDER BY のカラムにインデックスを追加
CREATE INDEX idx_orders_created_at ON orders (created_at DESC);
AP-4 SELECT * の多用

不要なカラムまで取得するとネットワーク転送量が増え、カバリングインデックスの恩恵も受けられない。カラム追加時に予期しない挙動の原因にもなる。

-- NG
SELECT * FROM users WHERE id = 1;

-- OK: 必要なカラムだけ明示する
SELECT id, name, email FROM users WHERE id = 1;
AP-5 NOT IN / NOT EXISTS の使いすぎ

NOT IN はサブクエリにNULLが含まれると意図しない結果になることがある。NOT EXISTS の方が安全で、インデックスも使われやすい。

-- NG: NULLが混入すると全件0件になるバグが起きることがある
SELECT * FROM users
WHERE id NOT IN (
    SELECT user_id FROM orders
);

-- OK: NOT EXISTS の方が安全で高速なことが多い
SELECT * FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o
    WHERE o.user_id = u.id
);

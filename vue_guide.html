Vue 3 教材 — 基礎〜応用

Composition API・Router・Pinia まで。HTMLとJavaScriptの基礎知識がある前提。

Vueとは何か

Vue（ビュー）はUIを作るためのJavaScriptフレームワーク。「データが変わったら画面も自動で変わる」という仕組みを中心に設計されている。

生のJavaScriptでは「データを変更する→自分でDOMを探して書き換える」という2ステップが必要だが、Vueでは「データを変更するだけ」でいい。

✗ 生のJS — 自分でDOMを操作
// データを変えて
count = count + 1;
// 自分でDOMを探して書き換える
document.getElementById('count').textContent = count;
✓ Vue — データを変えるだけ
// データを変えるだけで画面が自動更新
count.value = count.value + 1;
// DOMの操作はVueが勝手にやる

.vueファイル（SFC）の構造

SFC（Single File Component）= 1つの .vue ファイルの中に template・script・style を全部まとめて書く形式。

<!-- MyComponent.vue -->

<script setup>
// ① ロジック（データ・関数・computed等）をここに書く
// setup属性をつけると、ここに書いたものが自動でtemplate側から使える
import { ref } from 'vue'
const count = ref(0)
</script>

<template>
  <!-- ② 画面のHTML構造をここに書く -->
  <button @click="count++">{{ count }}</button>
</template>

<style scoped>
/* ③ このコンポーネントだけに適用されるCSS */
/* scoped をつけると他のコンポーネントに影響しない */
</style>
script setup は Vue 3.2から推奨の書き方。これより前は setup() 関数の中に書く形式だったが、現在はほぼ script setup 一択でいい。

リアクティビティ — データと画面を連動させる

Vueの核心。「変わったら画面も変わる」状態にする仕組み。

ref — 単一の値をリアクティブにする

import { ref } from 'vue'

const count = ref(0)         // 数値
const name  = ref('')         // 文字列
const items = ref([])         // 配列もOK
const show  = ref(true)       // 真偽値

// script内では .value でアクセスする
count.value = 5
console.log(count.value)   // 5

// template内では .value 不要（自動で外される）
// <p>{{ count }}</p> ← これでOK

reactive — オブジェクトをリアクティブにする

import { reactive } from 'vue'

const user = reactive({
  name: '山田',
  age: 32,
})

// .value 不要。直接アクセスできる
user.name = '田中'
console.log(user.age)     // 32
refreactive
対象プリミティブ（数値・文字列・配列など何でも）オブジェクト専用
アクセスscript内は .value 、template内は不要.value 不要。直接アクセス
よく使う場面単独の値、フォームの入力値、カウンター複数のプロパティをまとめて管理するとき
現場での傾向迷ったらrefでOK設定オブジェクトやフォームデータなど

computed — 他の値から計算した値

import { ref, computed } from 'vue'

const price    = ref(1000)
const quantity = ref(3)

// priceやquantityが変わると自動で再計算される
const total = computed(() => price.value * quantity.value)

console.log(total.value)   // 3000
price.value = 2000
console.log(total.value)   // 6000（自動で再計算）
⚠ computedの中でデータを変更してはいけない computedは「読み取り専用の計算結果」。中でref.valueを書き換えたりすると無限ループや予期しない動作になる。

watch — 値の変化を監視して副作用を実行する

import { ref, watch, watchEffect } from 'vue'

const keyword = ref('')

// watch: 特定の値の変化を監視
watch(keyword, (newVal, oldVal) => {
  console.log(`${oldVal} → ${newVal}`)
  // APIを叩くなど副作用を実行する
})

// watchEffect: 内部で参照した値が変わったら自動で実行
watchEffect(() => {
  // keywordを参照しているので、keywordが変わるたびに実行される
  console.log('現在のキーワード:', keyword.value)
})
watchwatchEffect
監視対象の指定明示的に指定する自動検出（内部で参照した値）
初回実行しない（immediate:trueで変更可）する
oldVal取得できる取得できない
よく使う場面特定の値の変化に反応したいとき複数の値に依存する副作用

テンプレート構文

マスタッシュ構文 — 値を表示する

<template>
  <!-- {{ }} でJSの値を表示 -->
  <p>{{ message }}</p>
  <p>{{ count * 2 }}</p>       <!-- 式もOK -->

v-bind — 属性をデータに紐づける

<!-- v-bind:href または :href（省略形）-->
<a :href="url">リンク</a>
<img :src="imgPath" :alt="imgAlt">
<button :disabled="isLoading">送信</button>

<!-- クラスのバインド -->
<div :class="{ active: isActive, error: hasError }"></div>
<!-- isActive=true → class="active" がつく -->

v-on — イベントを紐づける

<!-- v-on:click または @click（省略形）-->
<button @click="handleClick">クリック</button>
<input  @input="handleInput">
<form   @submit.prevent="handleSubmit">  <!-- .prevent でpreventDefault -->

v-model — 双方向バインディング

<script setup>
const name = ref('')
</script>

<template>
  <!-- inputの値とnameが自動で同期する -->
  <input v-model="name">
  <p>入力値: {{ name }}</p>

  <!-- v-modelは :value + @input のショートカット -->
  <!-- 以下と同じ意味 -->
  <input :value="name" @input="name = $event.target.value">
</template>

v-if / v-show — 条件で表示切り替え

<!-- v-if: DOM自体を追加・削除する -->
<p v-if="isLoggedIn">ようこそ</p>
<p v-else-if="isGuest">ゲストです</p>
<p v-else>ログインしてください</p>

<!-- v-show: display:none で非表示にする（DOMは残る）-->
<p v-show="isVisible">表示切り替え</p>
v-ifv-show
仕組みDOMを追加・削除するdisplay:noneで隠す（DOMは残る）
初期コストfalseのときコンポーネントが作られない（軽い）常にDOMが作られる（初期コストあり）
切り替えコスト毎回DOMを作り直す（頻繁な切り替えに不向き）CSSだけなので速い
向いているケース初期表示が不要なもの（条件が滅多に変わらない）頻繁に表示・非表示を切り替えるもの

v-for — リストのループ

const items = ref([
  { id: 1, name: 'りんご' },
  { id: 2, name: 'みかん' },
])

<!-- :key は必須。リストの差分更新に使われる。一意なIDを使う -->
<ul>
  <li v-for="item in items" :key="item.id">
    {{ item.name }}
  </li>
</ul>
⚠ :key にはインデックス（0,1,2...）を使わない リストの順番が変わったときに意図しない再レンダリングが起きる。必ずデータ固有のIDを使う。

コンポーネント — UIを部品に分ける

コンポーネントは「再利用できるUIの部品」。ボタン・カード・フォームなどを独立したファイルに切り出す。

いつコンポーネントに切り出すか

切り出すべきとき切り出さなくていいとき
同じUIを複数箇所で使う（RecipeCardを一覧表示するなど）一箇所しか使わない単純なUI
1つのファイルが長くなって読みにくいファイルが短くて見通しが良い
独立したロジックを持つ（フォームのバリデーション等）親のデータをそのまま表示するだけ

props — 親から子へデータを渡す

<!-- RecipeCard.vue（子） -->
<script setup>
const props = defineProps({
  dish: {
    type: Object,
    required: true,           // 必須
  },
  showDetail: {
    type: Boolean,
    default: false,          // デフォルト値
  }
})
</script>

<template>
  <div>
    <h2>{{ props.dish.name }}</h2>
    <p v-if="showDetail">詳細情報</p>
  </div>
</template>

<!-- App.vue（親） -->
<RecipeCard :dish="selectedDish" :show-detail="true" />

emit — 子から親へイベントを通知する

<!-- ChildButton.vue（子） -->
<script setup>
const emit = defineEmits(['click', 'update:value'])

const handleClick = () => {
  emit('click', { timestamp: Date.now() }) // 値も一緒に送れる
}
</script>

<!-- Parent.vue（親） -->
<ChildButton @click="handleChildClick" />
⚠ propsは読み取り専用 — 子コンポーネントで直接書き換えてはいけない ✗ NG: props.config.value = 'xxx' ← propsを直接書き換える
✓ OK: emit('update:config', newValue) ← 変更を親に通知して親が更新する

propsを直接書き換えると「どこでデータが変わったか」が追跡できなくなり、バグの原因になる。Piniaを使うとこの問題をスマートに解決できる（後述）。

v-model をコンポーネントで使う

<!-- 子コンポーネントでv-modelを受け取る -->
<script setup>
const props = defineProps({ modelValue: Number })
const emit  = defineEmits(['update:modelValue'])

const handleInput = (e) => {
  emit('update:modelValue', Number(e.target.value))
}
</script>

<template>
  <input :value="modelValue" @input="handleInput">
</template>

<!-- 親側での使い方 -->
<NumberInput v-model="count" />
<!-- v-model は :modelValue="count" @update:modelValue="count=$event" の省略形 -->

slot — コンポーネントに穴を開ける

コンポーネントの中に「外から内容を差し込める穴」を作る仕組み。

<!-- Card.vue（子） -->
<template>
  <div class="card">
    <slot name="header"></slot>    <!-- 名前付きスロット -->
    <slot></slot>                  <!-- デフォルトスロット -->
    <slot name="footer"></slot>
  </div>
</template>

<!-- 親側での使い方 -->
<Card>
  <template #header><h2>タイトル</h2></template>
  <p>本文がここに入る</p>    <!-- デフォルトスロットへ -->
  <template #footer><button>閉じる</button></template>
</Card>

provide / inject — 深いネストでのデータ渡し

props をバケツリレーで何段も渡していくのを避けるための仕組み。ただし Pinia の方が管理しやすいので、グローバルなデータは Pinia を使うことが多い。

// 祖先コンポーネント（データを提供する側）
import { provide } from 'vue'
provide('theme', 'dark')

// 子孫コンポーネント（何段深くても受け取れる）
import { inject } from 'vue'
const theme = inject('theme')  // 'dark'

defineExpose — 子の値を親から参照する

// 子コンポーネント
const count = ref(0)
defineExpose({ count }) // 外から参照できるようにする

// 親コンポーネント
const childRef = ref(null)
// <Child ref="childRef" />
console.log(childRef.value.count) // 子のcountにアクセス

ライフサイクルフック

コンポーネントが「作られる→表示される→更新される→消える」という流れの各タイミングに処理を差し込む仕組み。

onBeforeMount
DOMに表示される直前
ほぼ使わない
onMounted
DOMに表示された直後
API取得・DOMの初期化・外部ライブラリの初期化
onBeforeUpdate
データ変更でDOMが更新される直前
ほぼ使わない
onUpdated
DOMが更新された直後
DOM更新後に何かしたいとき（多用は避ける）
onBeforeUnmount
コンポーネントが消える直前
イベントリスナーの解除・タイマーのクリア
onUnmounted
コンポーネントが消えた直後
後片付け
import { onMounted, onUnmounted } from 'vue'

// 最もよく使うパターン: onMounted でデータを取得する
const recipes = ref([])

onMounted(async () => {
  const res = await fetch('/api/recipes')
  recipes.value = await res.json()
})

// onUnmounted: 後片付け（メモリリーク防止）
const timer = setInterval(() => {}, 1000)
onUnmounted(() => clearInterval(timer))

nextTick — DOM更新後に処理する

データを変更した直後はDOMがまだ更新されていない。更新後のDOMを参照したい場合に使う。

import { nextTick } from 'vue'

const message = ref('')

const update = async () => {
  message.value = '更新しました'
  // この時点ではDOMはまだ古い

  await nextTick()
  // ここでDOMが更新済みになっている
  console.log(document.querySelector('p').textContent)
}

Vue Router — ページ遷移を管理する

SPAでURLとコンポーネントを紐づける仕組み。URLが変わっても実際にはページを読み込まず、表示するコンポーネントを切り替えるだけ。

# インストール
npm install vue-router@4
// src/router/index.js
import { createRouter, createWebHistory } from 'vue-router'
import Home    from '../views/Home.vue'
import Detail  from '../views/Detail.vue'

const routes = [
  { path: '/',            component: Home },
  { path: '/recipe/:id',  component: Detail },  // :id はパラメータ
]

const router = createRouter({
  history: createWebHistory(),  // HTMLHistoryモード（#なしURL）
  routes,
})

export default router
// main.js にルーターを登録
import router from './router'
createApp(App).use(router).mount('#app')
<!-- App.vue: ページコンポーネントの表示場所 -->
<template>
  <nav>
    <RouterLink to="/">ホーム</RouterLink>
    <RouterLink to="/recipe/1">レシピ1</RouterLink>
  </nav>
  <RouterView />  <!-- ここにページコンポーネントが表示される -->
</template>
// コンポーネント内でルーターを使う
import { useRouter, useRoute } from 'vue-router'

const router = useRouter()  // ページ遷移に使う
const route  = useRoute()   // 現在のURL情報を取得

// パラメータの取得
console.log(route.params.id)   // /recipe/5 → '5'
console.log(route.query.page)  // ?page=2 → '2'

// プログラムでページ遷移
router.push('/')
router.push({ path: '/recipe', params: { id: 5 } })

Pinia — グローバルな状態管理

複数のコンポーネントから共有したいデータを一箇所で管理する仕組み。「どのコンポーネントからでも読み書きできるリアクティブなデータ置き場」。

なぜPiniaが必要か

propsとemitでデータを渡す方法には限界がある。コンポーネントが深くなるほどバケツリレーが長くなり、子が親のpropsを直接書き換えたくなってしまう。

✗ props直接書き換え（アンチパターン）
// 子コンポーネント
const props = defineProps({ config: Object })

// これは動くが正しくない
props.config.value = 'new'
// どこで変更されたか追跡できない
✓ Piniaで解決
// どのコンポーネントからでも
const store = useConfigStore()

// storeのactionを通して変更する
store.updateValue('new')
// 変更が一箇所で管理される
# インストール
npm install pinia
// main.js
import { createPinia } from 'pinia'
createApp(App).use(createPinia()).use(router).mount('#app')
// src/stores/recipeStore.js — storeの定義
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useRecipeStore = defineStore('recipe', () => {
  // --- state（データ）---
  const selectedKey = ref('')
  const recipes      = ref([])

  // --- getters（computedと同じ）---
  const selectedRecipe = computed(() =>
    recipes.value.find(r => r.name === selectedKey.value)
  )

  // --- actions（データを変更するメソッド）---
  const selectRecipe = (key) => {
    selectedKey.value = key
  }

  const fetchRecipes = async () => {
    const res = await fetch('/api/recipes')
    recipes.value = await res.json()
  }

  return { selectedKey, recipes, selectedRecipe, selectRecipe, fetchRecipes }
})
// コンポーネントでの使い方
import { useRecipeStore } from '../stores/recipeStore'

const store = useRecipeStore()

// データを読む
console.log(store.selectedKey)
console.log(store.selectedRecipe)

// データを変更する（actionを呼ぶ）
store.selectRecipe('カレー')

// storeDeconstruct（分割代入）はリアクティビティが失われるので注意
// NG: const { selectedKey } = store   ← 変化を追わなくなる
// OK: storeToRefs を使う
import { storeToRefs } from 'pinia'
const { selectedKey, selectedRecipe } = storeToRefs(store)
Piniaを使うと「どこでデータが変わったか」がstoreのactionに集約されるので、デバッグが楽になる。Vue DevToolsと組み合わせると状態の変化が可視化できる。

Options API vs Composition API

古いコードや他人のコードを読むときにOptions APIが出てくることがある。

Options API（Vue 2スタイル）
export default {
  data() {
    return { count: 0 }
  },
  computed: {
    double() { return this.count * 2 }
  },
  methods: {
    increment() { this.count++ }
  },
  mounted() {
    // 初期化処理
  }
}
Composition API（Vue 3 / 現在の標準）
<script setup>
const count = ref(0)

const double = computed(() => count.value * 2)

const increment = () => count.value++

onMounted(() => {
  // 初期化処理
})
</script>
Options APIComposition API
データdata()ref / reactive
算出プロパティcomputed: {}computed()
メソッドmethods: {}普通の関数
ライフサイクルmounted() などonMounted() など
TypeScript対応やや難しいしやすい
ロジックの再利用Mixinで複雑になりがちComposable関数として切り出せる

Composable — ロジックを再利用する

コンポーネントをまたいで使いたいロジックを関数として切り出す仕組み。慣習として use〇〇 という名前にする。

// src/composables/useCounter.js
import { ref } from 'vue'

export function useCounter(initialValue = 0) {
  const count = ref(initialValue)

  const increment = () => count.value++
  const decrement = () => count.value--
  const reset     = () => count.value = initialValue

  return { count, increment, decrement, reset }
}

// どのコンポーネントでも使える
const { count, increment } = useCounter(10)

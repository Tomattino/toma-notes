クエリを読み解き、設計を洗練させるための知識

EXPLAIN実践 — スロークエリを特定して改善する

EXPLAINは「このSQLをDBがどう実行するか」の計画を表示する。インデックスが効いているか、フルスキャンが起きていないかを確認するための必須ツール。

スロークエリ改善の流れ

スロークエリログの有効化（MySQL）

-- スロークエリログを有効にする（2秒以上かかったクエリを記録）
SET GLOBAL slow_query_log       = 'ON';
SET GLOBAL long_query_time      = 2;       -- 2秒以上を記録
SET GLOBAL slow_query_log_file  = '/var/log/mysql/slow.log';

-- インデックスを使わないクエリも記録する（開発時に便利）
SET GLOBAL log_queries_not_using_indexes = 'ON';

実際のEXPLAIN出力を読む

-- 問題のあるクエリ例
EXPLAIN SELECT
    u.name,
    COUNT(o.id) AS order_count
FROM       users  u
INNER JOIN orders o ON u.id = o.user_id
WHERE    YEAR(u.created_at) = 2024  -- ← SARGable違反
GROUP BY u.id, u.name;
-- EXPLAINの出力イメージ
-- id | type | key    | rows    | Extra
-- 1  | ALL  | NULL   | 1000000 | Using where; Using temporary; Using filesort
--    ↑フルスキャン  ↑インデックス未使用  ↑一時テーブル+filesortで最悪の状態
Before — type=ALL, rows=100万 
SELECT u.name, COUNT(o.id) AS cnt
FROM       users  u
INNER JOIN orders o ON u.id = o.user_id
WHERE    YEAR(u.created_at) = 2024
GROUP BY u.id, u.name;
After — type=range, rows=数千（インデックスで絞れる） 
-- ① SARGableに書き直す（関数をやめて範囲指定に）
-- ② orders.user_id にインデックスを追加
CREATE INDEX idx_orders_user_id   ON orders (user_id);
CREATE INDEX idx_users_created_at ON users  (created_at);

SELECT u.name, COUNT(o.id) AS cnt
FROM       users  u
INNER JOIN orders o ON u.id = o.user_id
WHERE    u.created_at >= '2024-01-01'
  AND  u.created_at <  '2025-01-01'
GROUP BY u.id, u.name;

EXPLAIN ANALYZE — 実際の実行時間を計測する（PostgreSQL）

-- EXPLAIN ANALYZEは実際にSQLを実行して計測する（本番DBでは注意）
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';

-- 出力例:
-- Index Scan using idx_users_email on users
--   Index Cond: (email = 'user@example.com')
--   Planning Time: 0.1 ms
--   Execution Time: 0.05 ms  ← 実際の実行時間
⚠ EXPLAIN ANALYZEはUPDATE/DELETEにも使えるが実際に実行される 本番DBで UPDATE や DELETE に EXPLAIN ANALYZE を使うと実際にデータが変更される。必ずSELECTか、トランザクション内でROLLBACKできる状態で使う。PostgreSQL固有

統計情報とANALYZE

DBオプティマイザはテーブルの統計情報（行数・値の分布等）をもとに実行計画を決める。統計情報が古いと最適でない計画が選ばれることがある。

-- MySQL: テーブルの統計情報を更新する
ANALYZE TABLE users;
ANALYZE TABLE orders;

-- PostgreSQL: 統計情報を更新する
ANALYZE users;

-- 現在のテーブル統計を確認（MySQL）
SELECT
    table_name,
    table_rows,
    data_length,
    index_length
FROM   information_schema.tables
WHERE  table_schema = 'your_database'
ORDER BY data_length DESC;
大量のINSERT/DELETE後はANALYZEを実行すると実行計画が改善されることがある。多くのDBは自動で統計情報を更新するが、急激にデータが変化した場合は手動実行が有効。

CTE（WITH句） — クエリを分かりやすく分割する

CTE（Common Table Expression）はWITH句で名前付きの一時的なクエリ結果を定義する機能。複雑なサブクエリを名前で参照できるようになり、可読性が大幅に上がる。

✗ サブクエリのネストで読みにくい
SELECT u.name, ds.total
FROM users u
INNER JOIN (
    SELECT
        user_id,
        SUM(amount) AS total
    FROM     orders
    WHERE    status = 'completed'
    GROUP BY user_id
    HAVING   SUM(amount) > 10000
) ds ON u.id = ds.user_id;
✓ CTEで名前をつけて読みやすくする
WITH heavy_buyers AS (
    SELECT
        user_id,
        SUM(amount) AS total
    FROM     orders
    WHERE    status = 'completed'
    GROUP BY user_id
    HAVING   SUM(amount) > 10000
)
SELECT u.name, hb.total
FROM       users       u
INNER JOIN heavy_buyers hb ON u.id = hb.user_id;

複数のCTEを組み合わせる

-- カンマで区切って複数定義できる
WITH
completed_orders AS (
    SELECT user_id, SUM(amount) AS total
    FROM     orders
    WHERE    status = 'completed'
    GROUP BY user_id
),
heavy_buyers AS (
    SELECT * FROM completed_orders  -- 前のCTEを参照できる
    WHERE  total > 10000
)
SELECT
    u.name,
    hb.total
FROM       users       u
INNER JOIN heavy_buyers hb ON u.id = hb.user_id
ORDER BY   hb.total DESC;

再帰CTE — 階層構造を扱う

親子関係・組織図・カテゴリツリーなど、自己参照するデータを扱うときに使う。

-- 例: カテゴリの階層構造（親カテゴリ→子カテゴリ→孫カテゴリ）
-- categories テーブル: id, name, parent_id

WITH RECURSIVE category_tree AS (
    -- ① アンカー（起点）: 最上位カテゴリ（parent_id が NULL）
    SELECT
        id,
        name,
        parent_id,
        0 AS depth,             -- 階層の深さ
        name AS path             -- パス文字列
    FROM  categories
    WHERE parent_id IS NULL

    UNION ALL

    -- ② 再帰（自分自身を参照）: 子カテゴリを順番に展開
    SELECT
        c.id,
        c.name,
        c.parent_id,
        ct.depth + 1,
        ct.path || ' > ' || c.name
    FROM       categories   c
    INNER JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT depth, name, path
FROM   category_tree
ORDER BY path;
-- 出力イメージ
-- depth | name     | path
-- 0     | 電子機器  | 電子機器
-- 1     | パソコン  | 電子機器 > パソコン
-- 2     | ノートPC  | 電子機器 > パソコン > ノートPC
-- 2     | デスクトップ | 電子機器 > パソコン > デスクトップ
-- 1     | スマホ    | 電子機器 > スマホ
⚠ 再帰CTEには無限ループのリスクがある データに循環参照（A→B→C→A のような構造）があると無限ループになる。MySQLは最大再帰数の上限（デフォルト1000）があるが、PostgreSQLはCYCLE句で対策できる。データの整合性を確認してから使う。DB固有

CASE式 — SQLの条件分岐

CASE式はSQLの中で条件分岐を書く構文。SELECT句・WHERE句・ORDER BY句・GROUP BY句のどこでも使える。

単純CASE式と検索CASE式

-- 単純CASE式: 1つのカラムの値で分岐
SELECT
    name,
    status,
    CASE status
        WHEN 'active'   THEN '有効'
        WHEN 'inactive' THEN '無効'
        WHEN 'pending'  THEN '審査中'
        ELSE                  '不明'
    END AS status_label
FROM users;

-- 検索CASE式: 条件式で分岐（より柔軟）
SELECT
    name,
    age,
    CASE
        WHEN age < 20 THEN '未成年'
        WHEN age < 65 THEN '成人'
        ELSE               'シニア'
    END AS age_group
FROM users;

集計とCASE式の組み合わせ（ピボット）

縦持ちのデータを横持ちに変換するときによく使う。

-- orders テーブル: status（completed/pending/cancelled）ごとの件数を横並びに
SELECT
    COUNT(CASE WHEN status = 'completed'  THEN 1 END) AS completed_count,
    COUNT(CASE WHEN status = 'pending'    THEN 1 END) AS pending_count,
    COUNT(CASE WHEN status = 'cancelled' THEN 1 END) AS cancelled_count,
    SUM(CASE WHEN   status = 'completed'  THEN amount ELSE 0 END) AS completed_amount
FROM orders;

CASE式でORDER BYをカスタマイズ

-- ステータスの表示順をビジネスロジックに合わせる
SELECT * FROM orders
ORDER BY
    CASE status
        WHEN 'pending'    THEN 1  -- 審査中を最優先
        WHEN 'completed'  THEN 2
        WHEN 'cancelled' THEN 3
        ELSE 4
    END,
    created_at DESC;

VIEW — クエリに名前をつけて再利用する

よく使うクエリを仮想テーブルとして保存する機能。実データは持たず、参照するたびに元のSQLが実行される。

VIEWの作成と使い方

-- VIEWの作成: ユーザーとロールを結合した一覧
CREATE VIEW user_role_view AS
SELECT
    u.id,
    u.name,
    u.email,
    r.name  AS role_name
FROM       users       u
INNER JOIN user_roles  ur ON u.id      = ur.user_id
INNER JOIN roles       r  ON ur.role_id = r.id;

-- VIEWはテーブルと同じように使える
SELECT * FROM user_role_view WHERE role_name = 'admin';

-- VIEWの更新
CREATE OR REPLACE VIEW user_role_view AS
SELECT -- ... 新しいクエリ

-- VIEWの削除
DROP VIEW user_role_view;

VIEWを使うべき場面・使わない方がいい場面

使うべき場面使わない方がいい場面
複雑なJOINを毎回書かずに済むパフォーマンスが重要な処理（VIEWは毎回実行される）
特定のカラムだけを公開するアクセス制御VIEWを重ねて使う（VIEW on VIEWは特に遅くなる）
ビジネスロジックをSQLレベルで共通化する更新・削除が必要な場合（単純なVIEWでないと不可）
マテリアライズドビュー（Materialized View）はVIEWの結果を実際に保存する機能。PostgreSQLで使える。参照が多いが更新が少ないデータに有効。PostgreSQL固有
-- PostgreSQL: マテリアライズドビュー
CREATE MATERIALIZED VIEW order_summary AS
SELECT
    user_id,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM     orders
GROUP BY user_id;

-- データを最新化する（手動実行が必要）
REFRESH MATERIALIZED VIEW order_summary;

シーケンス — 連番を生成する

重複しない連番（1, 2, 3, ...）を生成する仕組み。主キーのAUTO_INCREMENTに使われているが、明示的に使うこともできる。

MySQLのAUTO_INCREMENT

-- MySQLでは AUTO_INCREMENT で自動採番
CREATE TABLE orders (
    id     INT PRIMARY KEY AUTO_INCREMENT,
    amount INT NOT NULL
);

-- 現在の AUTO_INCREMENT 値を確認
SHOW TABLE STATUS LIKE 'orders';
-- Auto_increment カラムに次の採番値が表示される

-- AUTO_INCREMENT の値をリセット（TRUNCATEすると自動でリセットされる）
ALTER TABLE orders AUTO_INCREMENT = 1;

PostgreSQLのSEQUENCE

-- PostgreSQLではSEQUENCEオブジェクトを明示的に作れる
CREATE SEQUENCE order_id_seq
    START     1       -- 開始値
    INCREMENT 1       -- 増分
    MINVALUE  1       -- 最小値
    NO MAXVALUE       -- 上限なし
    CACHE     1;      -- キャッシュ数（性能に影響）

-- テーブルのカラムに紐づける
CREATE TABLE orders (
    id     INT DEFAULT NEXTVAL('order_id_seq') PRIMARY KEY,
    amount INT NOT NULL
);

-- または SERIAL / GENERATED ALWAYS AS IDENTITY（より推奨）
CREATE TABLE orders (
    id     INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    amount INT NOT NULL
);

-- シーケンスの現在値を取得・操作
SELECT NEXTVAL('order_id_seq');  -- 次の値を取得（値が進む）
SELECT CURRVAL('order_id_seq');  -- 現在のセッションの最後の値
SELECT SETVAL('order_id_seq', 1000);  -- 値をリセット

シーケンスの注意点

注意点理由
連番に歯抜けが発生することがあるROLLBACKしてもシーケンスは戻らない。採番後にトランザクションが失敗すると番号が飛ぶ
最大値に達するとエラーになるINT型の上限（約21億）を超えるとエラー。BIGINTを使うか定期的に監視する
TRUNCATE後もシーケンスは進む（MySQLのみリセット）PostgreSQLはTRUNCATEしてもシーケンスはリセットされない
⚠ シーケンスの連番に「欠番があってはいけない」ケースは設計を見直す 会計の伝票番号など「欠番が許されない」要件がある場合、AUTO_INCREMENTやシーケンスでは実現できない。別途採番テーブルを用意してトランザクション内で管理する必要がある。

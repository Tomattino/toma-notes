ER図 基礎教材

エンティティ・リレーションシップ図の読み方・書き方・設計の考え方

はじめに — 基本用語

ER図
Entity Relationship Diagram。DBのテーブル構造と関係を図で表した設計図。
エンティティ
管理したい「もの」や「概念」。テーブルに相当する。例：ユーザー、ロール、権限。
属性
エンティティが持つ情報。テーブルのカラムに相当する。例：名前、メールアドレス。
リレーション
エンティティ同士の関係。「ユーザーはロールを持つ」のような紐づきを表す線。
カーディナリティ
リレーションの「何対何か」を表す。1対1・1対多・多対多の3種類が基本。
主キー（PK）
レコードを一意に識別するカラム。重複・NULLは不可。通常はidカラム。
外部キー（FK）
他のテーブルの主キーを参照するカラム。テーブル間の紐づけに使う。
中間テーブル
多対多の関係を実現するための橋渡しテーブル。2つのFKを持つ。

IE記法（カラス足記法）の読み方

ER図の線の端についている記号で「何対何の関係か」を表す。現場で最もよく使われる記法。

カーディナリティ — 何対何の関係か

1対1（one-to-one）
片方の1レコードが、もう片方の1レコードとだけ対応する。
例：users ↔ user_profiles
ユーザー1人につきプロフィールが1つ
1対多（one-to-many）
片方の1レコードが、もう片方の複数レコードと対応する。最もよく使う。
例：roles → user_roles
1つのロールを複数ユーザーが持てる
多対多（many-to-many）
両方のテーブルが互いに複数対応する。中間テーブルで実現する。
例：users ↔ roles
ユーザーは複数ロールを持てる
ロールも複数ユーザーに割り当て可能
多対多を中間テーブルで分解する理由
RDBでは1セルに複数値を入れられない。中間テーブルで「1対多×2」に分解することで実現する。
users(1) → user_roles(多) ← roles(1)

論理削除（soft delete）

物理削除（DELETE文で完全に消す）ではなく、「削除済みフラグ」や「削除日時」カラムを使って削除を表現する手法。

物理削除論理削除
やることDELETE文でレコードを消すdeleted_at に日時をセットする
データ完全になくなるDBには残る
復元できないdeleted_at を NULL に戻せばよい
デメリットやり直しがきかない全クエリに WHERE deleted_at IS NULL が必要
-- 論理削除テーブルの例（deleted_atカラムを追加）
CREATE TABLE users (
    id         INT      PRIMARY KEY AUTO_INCREMENT,
    name       VARCHAR(100) NOT NULL,
    email      VARCHAR(255) NOT NULL UNIQUE,
    created_at DATETIME DEFAULT NOW(),
    deleted_at DATETIME DEFAULT NULL  -- NULLなら有効、日時が入ってれば削除済み
);

-- 論理削除の実行
UPDATE users
SET   deleted_at = NOW()
WHERE id = 1;

-- 有効なユーザーだけ取得する（全クエリにこれが必要）
SELECT * FROM users
WHERE deleted_at IS NULL;
⚠ 安易に全テーブルに入れない 論理削除は便利だが、全クエリに WHERE deleted_at IS NULL を追加し忘れると削除済みデータが混入するバグになる。採用するかどうかは要件次第で判断する。

CREATE TABLE — 実際のSQL

外部キーが参照する親テーブルを先に作る必要がある。中間テーブルは最後。

-- 1. 親テーブルから作る
CREATE TABLE roles (
    id          INT          PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(50)  NOT NULL UNIQUE,
    description TEXT,
    created_at  DATETIME     DEFAULT NOW()
);

CREATE TABLE permissions (
    id          INT          PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(100) NOT NULL UNIQUE,
    description TEXT,
    created_at  DATETIME     DEFAULT NOW()
);

CREATE TABLE users (
    id            INT          PRIMARY KEY AUTO_INCREMENT,
    name          VARCHAR(100) NOT NULL,
    email         VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    created_at    DATETIME     DEFAULT NOW()
);

-- 2. 中間テーブルは最後（両方の親が必要なので）
CREATE TABLE user_roles (
    id          INT      PRIMARY KEY AUTO_INCREMENT,
    user_id     INT      NOT NULL,
    role_id     INT      NOT NULL,
    assigned_at DATETIME DEFAULT NOW(),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE
);

CREATE TABLE role_permissions (
    id            INT  PRIMARY KEY AUTO_INCREMENT,
    role_id       INT  NOT NULL,
    permission_id INT  NOT NULL,
    FOREIGN KEY (role_id)       REFERENCES roles(id)       ON DELETE CASCADE,
    FOREIGN KEY (permission_id) REFERENCES permissions(id) ON DELETE CASCADE
);

このER図を使ったSQLの例

-- ユーザーID=1 が持つ権限を全部取得
SELECT
    u.name           AS user_name,
    r.name           AS role_name,
    p.name           AS permission_name
FROM            users            u
INNER JOIN      user_roles       ur ON u.id  = ur.user_id
INNER JOIN      roles            r  ON ur.role_id = r.id
INNER JOIN      role_permissions rp ON r.id  = rp.role_id
INNER JOIN      permissions      p  ON rp.permission_id = p.id
WHERE u.id = 1
ORDER BY r.name, p.name;

-- 「user:delete」権限を持つユーザーだけ取得
-- ※ permission.name を文字列で判定するのは現場でも一般的
--   idで判定するとコードを読んだときに何の権限か分からなくなりやすい
SELECT DISTINCT
    u.id,
    u.name,
    u.email
FROM       users            u
INNER JOIN user_roles       ur ON u.id         = ur.user_id
INNER JOIN roles            r  ON ur.role_id   = r.id
INNER JOIN role_permissions rp ON r.id         = rp.role_id
INNER JOIN permissions      p  ON rp.permission_id = p.id
WHERE p.name = 'user:delete';

正規化 — データの冗長性を排除する設計原則

同じ情報を複数箇所に持つと、更新漏れや矛盾が起きやすくなる。正規化はそれを防ぐための設計ルール。

正規化の前に知っておく概念：「依存」とは

正規化を理解する上で「〇〇は△△によって決まる」という関係を「△△に依存する」と表現する。

例依存の関係
社員番号がわかれば社員名が決まる社員名は社員番号に依存する
商品IDがわかれば商品名が決まる商品名は商品IDに依存する
部署IDがわかれば部署名が決まる部署名は部署IDに依存する

正規化は「どのカラムが何に依存しているか」を整理してテーブルを分割していく作業。

第1〜第3正規形の違い（全体像）

正規形一言で言うと解決する問題
第1正規形（1NF）1セルに1値だけセルに複数値や繰り返しグループが入っている
第2正規形（2NF）主キー全体に依存させる複合主キーの「一部」だけに依存するカラムがある
第3正規形（3NF）主キー以外への依存を排除する主キーではなく「別の普通のカラム」に依存するカラムがある

第2と第3の違いは「どこへの依存が問題か」の違い。第2は「主キーの一部への依存」が問題、第3は「主キー以外の普通のカラムへの依存」が問題。

第1正規形（1NF） 1つのセルに1つの値だけ入れる
ルール: 1つのセルに複数の値を詰め込まない・同じ種類のカラムを横に並べない

違反パターンは2種類ある。

✗ NG パターン1 — 1つのセルに複数値を詰め込む

user_id（PK）namerole_ids ← 問題
1山田1,2,3
2田中2

「role_id=2 のユーザーを全員取得」するSQLが書けない。更新・削除も困難。

✗ NG パターン2 — 繰り返しグループ（同じ種類のカラムを横に並べる）

user_id（PK）namerole_id_1role_id_2role_id_3
1山田123
2田中2NULLNULL

4つ目のロールが必要になったとき、カラムを追加しないといけない。NULLだらけになる。これが「繰り返しグループ」。

✓ OK — 中間テーブルで1行1値に分解する

id（PK）user_id（FK）role_id（FK）
111
212
313
422

1行に1つの紐づきだけを持つ。ロールがいくつ増えても行を追加するだけでよい。

第2正規形（2NF） 主キーの「一部」だけに依存するカラムを追い出す
問題: 複合主キー（2つ以上のカラムで1行を識別するとき）の一部だけで値が決まるカラムがある

第2正規形が問題になるのは複合主キーがある場合だけ。主キーが1カラムなら第2正規形違反は起きない。

具体例で説明する。受注明細テーブルは「どの注文のどの商品か」を特定するために order_id と product_id の2つで主キーを構成する（複合主キー）。

✗ NG — 受注明細テーブルに商品名を持たせている

order_id（PK）product_id（PK）quantityproduct_name ← 問題
11013ノートPC
11021マウス
21012ノートPC

product_name は product_id だけで決まる（「101番の商品名はノートPC」はどの注文でも同じ）。order_id は関係ない。
つまり主キーが order_id + product_id の2つなのに、product_name は product_id 1つだけに依存している。これが「主キーの一部への依存（部分関数従属）」。

何が困るか：「ノートPC」→「ノートパソコン」に名前が変わったとき、この商品が含まれる全注文行を更新しないといけない。更新漏れが起きると同じ商品なのに「ノートPC」と「ノートパソコン」が混在する。

✓ OK — product_name を products テーブルに切り出す

order_details（受注明細）

order_id（PK）product_id（PK）quantity
11013
11021
21012

products（商品）

id（PK）name
101ノートPC
102マウス

product_id に依存するカラム（商品名）は products テーブルにまとめる。名前変更は products の1行を更新するだけで済む。

第3正規形（3NF） 普通のカラムが別の普通のカラムに依存していたら追い出す
問題: 主キーではない普通のカラムが、さらに別の普通のカラムに依存している

第2正規形は「主キーの一部への依存」が問題だったが、第3正規形は「主キーと全く関係ない普通のカラムへの依存」が問題。

具体例で説明する。usersテーブルに部署名を直接持たせたケース。

✗ NG — users に department_name を直接持たせている

id（PK）namedepartment_iddepartment_name ← 問題
1山田10開発部
2田中10開発部
3鈴木20営業部

department_name は department_id によって決まる（「10番の部署は開発部」はどのユーザーでも同じ）。主キーである id は関係ない。
つまり主キー（id）ではなく、普通のカラムである department_id に依存している。これが「主キー以外への依存（推移的関数従属）」。

何が困るか：「開発部」→「開発グループ」に名称変更するとき、開発部に所属する全ユーザー行を更新しないといけない。更新漏れが起きると「開発部」と「開発グループ」が混在する矛盾が発生する。

主キーが複合かどうか複合主キーが前提主キーが1つでも起きる
解決策依存先を主キーにしたテーブルに切り出す依存先を主キーにしたテーブルに切り出す

名称変更は departments の1行を更新するだけで済む。usersは department_id（FK）で参照するだけ。

実務では「第3正規形まで正規化してから、パフォーマンス上の理由で意図的に非正規化する」という流れが基本。最初から非正規化するのは設計ミスの温床になる。
補足 第3正規形違反はどういうときに起きるか

教材のNG例は department_id と department_name を同じテーブルに共存させているが、最初からIDを振って設計した場合はこの状態は自然に起きにくい。実際の違反パターンは主に2つある。

パターン1 — 最初からIDなしで設計した

users
  id
  name
  department_name　← IDなしで文字列を直接持たせた

「部署は数個しかないから文字列で直接持てばいい」という判断で設計したケース。後から部署が増えたり名称変更が起きたりしたときに全行更新が必要になって困る。

パターン2 — FKはあるが、JOINが面倒で情報を後から追加した

users
  id
  name
  department_id　← FKとして元々あった（これはOK）
  department_name　← 「JOINが面倒だから」と後から追加（NG）

JOINを書く手間を省くために、よく使うカラムをそのテーブルに直接追加してしまうケース。IDが設計に入っていても、運用の中で違反が持ち込まれることがある。

正しい状態のカラム構成

users
  id　　　　　　　← 主キー
  name　　　　　　← id に依存（OK）
  department_id　← id に依存（OK）。FKとして必要
  department_name　← department_id に依存（NG）← 追い出す

department_id 自体は問題ない。「ユーザーがどの部署に所属しているか」という情報として必要なFK。問題なのは department_name で、これは department_id があれば departments テーブルをJOINすれば取れるものをわざわざ重複して持っている状態。

実務でよく見る第3正規形違反の例：orders テーブルに customer_name を直接持つ（customer_id があれば customers からJOINで取れる）、order_items に product_price を直接持つ（products から取れる）など。ただし「注文時点の価格を履歴として保持したい」という意図がある場合は意図的な非正規化として許容されることもある。

上級者向け — 設計で判断が必要なポイント

NULL許容の設計判断

状況設計理由
必ず入力されるべき値NOT NULLアプリ側だけでなくDB側でも強制する
任意の項目NULL許容NULLは「未入力」を表す。空文字とは別物。
FKカラム（必須の紐づき）NOT NULL例：注文の user_id（注文には必ずユーザーが存在する）
FKカラム（任意の紐づき）※1NULL許容例：記事の approved_by（未承認の記事は承認者がいない）

※1 任意の紐づき = 「あっても なくてもよい」関係。今回の例で言うと、ユーザー作成直後はまだロールが割り当たっていない場合があるケースなど。FKがNULLだと「まだ割り当てられていない」という意味になる。

複合主キー vs サロゲートキー（id列）

中間テーブルの主キーをどう設計するか。2つのアプローチがある。

複合主キーサロゲートキー（id列）
定義user_id + role_id を PK にする別途 id カラムを PK にする
重複防止自動で防げるUNIQUE制約を別途つける必要がある
JOIN少し複雑になるシンプル
現場の傾向古い設計に多い最近はこちらが主流

複合主キーの例

-- user_id + role_id の組み合わせがPK = 同じ組み合わせは必ず1件だけ
CREATE TABLE user_roles (
    user_id INT NOT NULL,
    role_id INT NOT NULL,
    PRIMARY KEY (user_id, role_id),  -- 2カラムを合わせてPK
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (role_id) REFERENCES roles(id)
);

サロゲートキー（今回の設計）の例

-- id列をPKにして、重複防止はUNIQUE制約で対応する
CREATE TABLE user_roles (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    role_id INT NOT NULL,
    UNIQUE (user_id, role_id),  -- 重複防止はUNIQUE制約で
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (role_id) REFERENCES roles(id)
);

ON DELETE とは

親テーブルのレコードを削除したとき、そのレコードを参照している子テーブルの行をどう扱うかを指定するオプション。

例えば users から user_id=1 を削除するとき、user_roles に user_id=1 の行が残っていたら参照先がなくなる（整合性が崩れる）。これをどう処理するかの設定。

オプション動作使いどころ
CASCADE親を削除したら子も自動で削除中間テーブルや親に従属するデータ。ユーザー削除時に user_roles も一緒に消したい場合など
RESTRICT子が存在する場合は親の削除を禁止（エラーになる）誤削除を防ぎたい重要データ。注文が残っているユーザーは消せないようにするなど
SET NULL子のFKカラムをNULLに更新する紐づきだけ外してデータは残したい場合。承認者が退職したが記事は残したいなど
NO ACTION何もしない（DBによりRESSTRICTと同じ動作）アプリ側で削除順序を制御する場合
-- CASCADE: ユーザー削除時にuser_rolesも自動で消える
FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE

-- RESTRICT: 注文が残っているユーザーは削除できない
FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE RESTRICT

-- SET NULL: 承認者が消えても記事は残す（approved_byはNULL許容が必要）
FOREIGN KEY (approved_by) REFERENCES users(id) ON DELETE SET NULL

インデックスとはどうして速くなるのか

インデックスは「本の索引（インデックスページ）」と同じ仕組み。

インデックスなしで検索する場合、DBは先頭から全行を順番にスキャンする（フルスキャン）。100万行あれば100万行全部見る。

インデックスはB-tree（バランス木）という構造でデータを管理している。「9999を探せ」と言われたら、まず「50万より小さいか」→「25万より小さいか」→「1万より大きいか」と二分探索で絞り込んでいく。100万行あっても20回程度の比較で見つかる。

-- FKカラムへのインデックス（JOINやWHEREで頻繁に使うカラム）
CREATE INDEX idx_user_roles_user_id ON user_roles (user_id);
CREATE INDEX idx_user_roles_role_id ON user_roles (role_id);

-- 複合インデックス: 重複防止も兼ねる
CREATE UNIQUE INDEX idx_user_roles_unique ON user_roles (user_id, role_id);
⚠ インデックスを闇雲につけると逆に遅くなる インデックスはSELECT（検索）を速くするが、INSERT・UPDATE・DELETE のたびにインデックスも更新するコストがかかる。書き込みが多いテーブルに大量のインデックスをつけると書き込み処理が遅くなる。よく使う検索条件（WHERE句・JOIN・ORDER BY）のカラムに絞ってつけるのが基本。

複合主キー vs サロゲートキーとON DELETEの判断まとめ

シーン推奨
中間テーブルのPKサロゲートキー（id）+ UNIQUE制約が最近の主流
ユーザー削除→紐づくデータも消えてほしいON DELETE CASCADE
重要データの誤削除を防ぎたいON DELETE RESTRICT
親が消えても子のデータは残したいON DELETE SET NULL（FKはNULL許容が必要）

ER図の読み方まとめ

見るポイント確認すること
線の端の記号縦線=1、カラス足=多。どちら側が1でどちらが多か確認する
PK（主キー）何でレコードを一意に識別するか
FK（外部キー）どのテーブルのどのカラムを参照しているか
中間テーブル多対多の関係を実現している。必ず2つのFKを持つ
NOT NULL / NULL許容そのカラムが必須かどうか
UNIQUE制約重複を許さないカラム（メールアドレス・ロール名など）
ER図を見たらまず「テーブル数と中間テーブルの有無」を確認する。中間テーブルがあれば多対多の関係がある。次にPK→FKの流れを追って「どのテーブルがどのテーブルに依存しているか」を把握すると全体の構造が見えてくる。

SQL 中級 — トランザクション・ロック編

データの整合性を守る仕組みと、並行処理の落とし穴

トランザクションとは

複数のSQL操作を「一塊の処理」として扱う仕組み。全部成功するか、全部なかったことになるかの2択にできる。

例：銀行の送金。「A口座から1万円引く」と「B口座に1万円加える」の2つは必ずセットで成功しないといけない。片方だけ成功すると1万円が消える。

ACID特性 — トランザクションが保証する4つの性質

性質意味具体例
Atomicity（原子性）全部成功か全部失敗か。中途半端な状態にならない送金の途中でエラーになっても、引き落としだけが残らない
Consistency（一貫性）トランザクション前後でデータが整合している送金前後で両口座の合計残高が変わらない
Isolation（独立性）並行するトランザクションが互いに干渉しない自分の処理中に他の人の処理が混入しない
Durability（永続性）コミットしたデータはシステム障害が起きても消えない確定した送金はサーバー再起動後も残る

基本構文

BEGIN;  -- トランザクション開始（START TRANSACTION とも書く）

-- ここに複数の操作を書く
UPDATE accounts SET balance = balance - 10000 WHERE id = 1;
UPDATE accounts SET balance = balance + 10000 WHERE id = 2;

COMMIT;    -- 全部確定（ここで初めてDBに永続的に反映される）
-- ROLLBACK; -- 全部なかったことにする
COMMITするまでは他のセッションからは変更が見えない（分離レベルによる）。COMMIT前ならROLLBACKでいつでも取り消せる。

ロールバック — 失敗したときに巻き戻す

処理の途中でエラーが起きたとき、それまでの変更をまとめて取り消す。アプリ側でエラーハンドリングと組み合わせて使う。

正常系と異常系のパターン

-- 正常系: 両方のUPDATEが成功したらCOMMIT
BEGIN;

UPDATE accounts
SET   balance = balance - 10000
WHERE id = 1 AND balance >= 10000; -- 残高不足チェック

-- アプリ側で影響行数を確認（0行ならROLLBACK）
-- affected_rows = 0 → ROLLBACK して「残高不足」エラーを返す

UPDATE accounts
SET   balance = balance + 10000
WHERE id = 2;

COMMIT; -- 両方成功したら確定
-- 異常系: エラー発生時にROLLBACKで全部取り消す
BEGIN;

UPDATE accounts SET balance = balance - 10000 WHERE id = 1;
-- ここでDBエラーやアプリエラーが発生

ROLLBACK; -- ↑のUPDATEもなかったことになる
-- id=1 の残高は元の値のまま

SAVEPOINTで部分的にロールバックする

大きなトランザクションの中で「ここまでは確定、ここからだけ取り消したい」というときに使う。

BEGIN;

INSERT INTO orders (user_id, amount) VALUES (1, 5000);

SAVEPOINT after_order; -- ここにセーブポイントを作る

INSERT INTO order_items (order_id, product_id) VALUES (1, 101);
-- order_itemsの登録だけ失敗したとする

ROLLBACK TO after_order; -- order_itemsの登録だけ取り消し（ordersは残る）

-- リトライ処理などを入れてからCOMMIT
COMMIT;
⚠ ROLLBACKはDDL（CREATE/DROP/ALTER/TRUNCATE）には効かない場合が多い MySQLではDDLは自動的にCOMMITされる。TRUNCATEもROLLBACKできないDBが多い。変更が大きいスキーマ操作は必ずバックアップを先に取る。DB固有

トランザクション分離レベル

並行するトランザクションが互いにどの程度影響し合うかを制御する設定。安全性と性能のトレードオフがある。

分離レベルで発生しうる問題

問題名何が起きるか具体例
ダーティリードまだCOMMITされていない（取り消されるかもしれない）他トランザクションのデータを読んでしまうAが残高を90000に変更中（未COMMIT）。BがSELECTすると90000が見える。その後AがROLLBACKすると、Bが見た90000は存在しないデータだった
ノンリピータブルリード同じSELECTを2回実行したら、間に別トランザクションがCOMMITして結果が変わってしまうAがid=1の残高を読んで100000。次の処理で再度読んだら90000になっていた（BがCOMMITしたため）。同じトランザクション内で値が変わる
ファントムリード同じ条件のSELECTを2回実行したら、間に別トランザクションがINSERTして行数が増えてしまうAが「残高10万以上の口座」を集計して3件。次に再集計したら4件になっていた（Bが新しい口座をINSERTしてCOMMITしたため）

分離レベルと防げる問題

分離レベルダーティリードノンリピータブルリードファントムリード性能
READ UNCOMMITTED発生する発生する発生する最速
READ COMMITTED防げる発生する発生する速い
REPEATABLE READ防げる防げる発生する※普通
SERIALIZABLE防げる防げる防げる遅い

※ MySQLのREPEATABLE READはMVCC（後述）によりファントムリードも防げる場合が多い。

-- 分離レベルの確認と変更（MySQL）
SELECT @@transaction_isolation;  -- 現在の分離レベルを確認

SET SESSION transaction_isolation = 'READ-COMMITTED';  -- セッション単位で変更
SET GLOBAL  transaction_isolation = 'REPEATABLE-READ'; -- サーバー全体で変更
MySQLのデフォルトはREPEATABLE READ。PostgreSQLのデフォルトはREAD COMMITTED。多くのWebアプリはデフォルトのまま使って問題ない。DB固有

なぜ分離レベルで防げるのか

「読み取りを禁止する」のではなく、「どのタイミングのデータを見せるか」を制御している。

分離レベルどう防いでいるか
READ COMMITTED「COMMITされたデータだけを返す」というルールで動く。COMMITされていないデータは存在しないものとして扱う。ダーティリードが防げるのはこのため
REPEATABLE READ「このトランザクションが始まった時点のスナップショットを見せ続ける」というルールで動く。他がCOMMITしても、自分のトランザクション中は最初に見たデータが返ってくる
SERIALIZABLE読み取り対象の行全体にロックをかけて、他のトランザクションの変更・追加をブロックする。最も安全だが最も遅い

ロック待機中のクエリはどうなる？

ロックが取れないとき、クエリは基本的に保留（待機）になる。エラーにはならず、ロックが解放されるまでじっと待つ。

状況動作
通常のロック待機保留（待機）。ロック解放後に自動的に処理が続く
タイムアウト設定あり（innodb_lock_wait_timeout）設定時間が過ぎるとエラーが返る。デフォルトは50秒
デッドロック検出時DBが自動検出して一方を強制的にROLLBACKしエラーを返す。これだけは即時エラー

「遅い」の目安

ロックやトランザクションで処理が遅くなるとは具体的にどのくらいか。Webアプリでの目安：

応答時間体感
〜100ms快適。ほぼ待ち時間を感じない
100〜500ms許容範囲。少し待つ感覚がある
500ms〜1秒体感で遅い。ユーザーが「重いな」と感じ始める
1秒以上明確に遅い。離脱・不満につながる

MySQLのlong_query_timeデフォルトは10秒だが、実務では0.5〜2秒に設定することが多い。バッチ処理は数分かかっても許容される場合がある。

transaction_isolation とは

DBの動作を変えるためのシステム変数のひとつ。カラムではなく「DBサーバ全体の設定項目」。innodb_lock_wait_timeout（ロックタイムアウト秒数）やmax_connections（最大接続数）なども同じ種類の設定変数で、SHOW VARIABLES;で全一覧を確認できる。

スコープ意味特徴
SESSION自分が接続している間だけ有効接続を切ると元の設定に戻る。他のユーザーに影響しない
GLOBALそのDBサーバに接続している全員に影響サーバ再起動後も設定が残る（my.cnfに書いた場合）。本番での変更は慎重に

ロック — 同時アクセスの衝突を防ぐ

複数のトランザクションが同じデータを同時に変更しようとしたときの衝突を防ぐ仕組み。

共有ロックと排他ロック

種類別名他の共有ロックと共存他の排他ロックと共存用途
共有ロック（S Lock）読み取りロックできるできないSELECT FOR SHARE
排他ロック（X Lock）書き込みロックできないできないSELECT FOR UPDATE / UPDATE / DELETE
-- 排他ロック: 他のトランザクションがUPDATEできなくなる
BEGIN;
SELECT * FROM accounts
WHERE id = 1
FOR UPDATE; -- このレコードに排他ロックをかける

-- この間、他のトランザクションはid=1をUPDATEしようとすると待機させられる

UPDATE accounts SET balance = balance - 10000 WHERE id = 1;
COMMIT; -- COMMITするとロックが解放される

-- 共有ロック: 読み取りはできるが更新はブロック
SELECT * FROM accounts WHERE id = 1 FOR SHARE;

「共有ロック同士は共存できる」= 複数のトランザクションが同時にSELECT FOR SHAREできる。読むだけなら衝突しないので。
ただし共有ロック中にUPDATEしようとすると待機させられる。共有ロックが全て解放されるまで排他ロックは取れない。「読んでいる途中に書き換えられる」は起きない。

楽観ロック vs 悲観ロック

ロックの戦略として2つのアプローチがある。

悲観ロック（Pessimistic Lock）

「衝突が起きるだろう」と想定して、最初からロックを取る。

BEGIN;
-- 読む時点でロックをかける
SELECT * FROM accounts
WHERE id = 1
FOR UPDATE;

UPDATE accounts
SET   balance = balance - 10000
WHERE id = 1;
COMMIT;

✓ 衝突が確実に防げる
✗ ロック中は他が待機→性能低下

楽観ロック（Optimistic Lock）

「衝突はめったに起きない」と想定してロックしない。更新時に変更されていないか確認する。

-- versionカラムで管理
SELECT id, balance, version
FROM  accounts WHERE id = 1;
-- version=5 を取得したとする

UPDATE accounts
SET
    balance = balance - 10000,
    version = version + 1
WHERE id      = 1
  AND version = 5; -- 変わってたら0行
-- 0行 → 他が更新済み → リトライ

✓ ロックしないので性能が高い
✗ 衝突時はアプリ側でリトライが必要

観点悲観ロック楽観ロック
向いているケース衝突が頻繁、金融系など整合性最優先衝突が稀、読み取り多め、性能優先
実装場所DB（FOR UPDATE）アプリ側（versionチェック）
待機ロック解放まで待機する待機なし（衝突したらリトライ）

デッドロック — お互いが待ち続ける状態

2つのトランザクションがお互いに相手のロックを待ち続けて、永遠に進めなくなる状態。

デッドロックが起きるパターン

時刻
トランザクションA
トランザクションB
T1
BEGIN
accounts id=1 をFOR UPDATE（ロック取得）
BEGIN
accounts id=2 をFOR UPDATE（ロック取得）
T2
accounts id=2 をFOR UPDATE しようとする
→ Bがロック中なので待機...
accounts id=1 をFOR UPDATE しようとする
→ Aがロック中なので待機...
T3
DBがデッドロックを検出してAをROLLBACK
Bのロックが解放されて処理続行
-- デッドロックを引き起こすコード例
-- トランザクションA: id=1 → id=2 の順でロック
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- (ここで少し処理)
SELECT * FROM accounts WHERE id = 2 FOR UPDATE; -- Bを待機

-- トランザクションB: id=2 → id=1 の順でロック（逆順！）
BEGIN;
SELECT * FROM accounts WHERE id = 2 FOR UPDATE;
-- (ここで少し処理)
SELECT * FROM accounts WHERE id = 1 FOR UPDATE; -- Aを待機 → デッドロック

デッドロックの回避策

対策内容効果
ロック取得順序を統一する常に id の小さい順（id=1→id=2）でロックを取得するようにコードを統一する最も効果的。根本的に解決できる
トランザクションを短くするロックを保持する時間を最小限にする。SELECT FOR UPDATE → すぐUPDATE → すぐCOMMITデッドロックの確率を下げる
ロックタイムアウトを設定する一定時間待ってもロックが取れなければエラーにする無限待機を防ぐ（根本解決ではない）
アプリ側でリトライするデッドロックエラーを検知したら処理をリトライするユーザー体験を損なわない（根本解決ではない）
-- デッドロック回避: ロック取得順序を統一する
-- AもBも「小さいidから順に」ロックを取る
BEGIN;
SELECT * FROM accounts
WHERE id IN (1, 2)
ORDER BY id  -- 必ず昇順で
FOR UPDATE;  -- id=1 → id=2 の順でロックされる

-- ロックタイムアウトの設定（MySQL）
SET innodb_lock_wait_timeout = 5; -- 5秒待ってもロックが取れなければエラー
⚠ デッドロックはDBが自動検出してROLLBACKする デッドロックが発生すると、DBはどちらかのトランザクションを強制的にROLLBACKしてエラーを返す。アプリ側でこのエラーを適切にハンドリングしてリトライする実装が必要。

MVCC — ロックなしで読み取りを速くする仕組み

MVCC（Multi Version Concurrency Control） = 多版型同時実行制御。データの複数のバージョンを保持することで、読み取りと書き込みがお互いをブロックしない仕組み。

InnoDB（MySQL）とPostgreSQLはどちらもMVCCを実装している。「SELECT（読み取り）がUPDATE（書き込み）をブロックしない」のはMVCCのおかげ。DB固有

レコードが増え続けないのか

増えるが、無限には増えない。不要になった古いバージョンは自動的に削除される。

DB古いバージョンの管理方法
MySQL（InnoDB）undo log という別領域に古いバージョンを保持する。参照しているトランザクションがなくなると自動削除される
PostgreSQLテーブル内にそのまま古い行が残る。VACUUM（バキューム）という処理が定期的に不要行を回収する。autovacuumが自動で実行する

古い情報を読んでしまわないか

意図的に古い情報を返す。REPEATABLE READは「トランザクション開始時点の情報を一貫して返す」のが目的なので、他がCOMMITしても自分の見えるデータは変わらない。これが正しい動作。

「今の最新値を読みたい」場合はどうするか：

方法内容
READ COMMITTEDに変更するSELECTするたびに最新のCOMMIT済みデータを返す
FOR UPDATEをかけるREPEATABLE READでも、FOR UPDATEを使うと最新値を取得できる

用途によって使い分ける。ほとんどのWebアプリはREPEATABLE READ（デフォルト）のままで問題ない。

実務でよくある実装パターン

在庫の二重販売を防ぐ（悲観ロック）

FOR UPDATEがない場合に何が起きるか：
AとBが同時にstock=1を読む → 両方「在庫あり」と判断 → 両方UPDATEを実行 → 在庫が-1になる（二重販売）

FOR UPDATEがある場合：
AがFOR UPDATEでロックを取得 → Bは待機 → AがUPDATE（stock=0）してCOMMIT → ロック解放 → Bがstock=0を読む → 「在庫なし」として処理を中断

BEGIN;

-- 在庫を確認しながらロックをかける
SELECT stock
FROM  products
WHERE id = 101
FOR UPDATE;  -- このレコードをロック。他のトランザクションはここで待機する

-- アプリ側で stock > 0 を確認してからUPDATE
UPDATE products
SET   stock = stock - 1
WHERE id    = 101
  AND stock > 0;  -- 念のためWHEREでも確認（0行なら在庫なし）

COMMIT;  -- ここでロック解放。待機していたBが処理を再開する

ポイント付与の重複防止（楽観ロック）

-- ポイントテーブルにversion列を持たせる
SELECT id, points, version
FROM  user_points
WHERE user_id = 1;
-- points=1000, version=3 を取得

UPDATE user_points
SET
    points  = 1100,       -- 1000 + 100ポイント付与
    version = 4           -- version をインクリメント
WHERE user_id = 1
  AND version  = 3;       -- 取得時と同じversionかチェック
-- affected_rows=0 なら他のリクエストが先に更新済み → リトライ

具体的に何が起きているか：

AとBが同時にversion=3のレコードを取得
→ A が先に UPDATE（version=3→4）して成功
→ B が UPDATE しようとするが WHERE version=3 にマッチする行がもうない → 0行
→ B は affected_rows=0 を検知してリトライ
→ B が再度SELECT → version=4 を取得
→ B が WHERE version=4 で UPDATE → 成功（version=4→5）

結果：AとBの更新が両方正しく反映される。楽観ロックなしだと両方がversion=3で更新してどちらかが上書きされる。

長時間のトランザクションを避ける

-- NG: 外部API呼び出しをトランザクション内に入れてしまう
BEGIN;
UPDATE orders SET status = 'processing' WHERE id = 1;
-- 外部決済APIを呼ぶ（数秒かかることもある）
-- この間ずっとロックを保持してしまう
UPDATE orders SET status = 'paid' WHERE id = 1;
COMMIT;

-- OK: 外部API呼び出しはトランザクションの外で行う
BEGIN;
UPDATE orders SET status = 'processing' WHERE id = 1;
COMMIT;  -- 一旦確定してロックを解放

-- 外部決済APIを呼ぶ（トランザクション外）

BEGIN;
UPDATE orders SET status = 'paid' WHERE id = 1;
COMMIT;
⚠ トランザクションを長くするとロックが増えて他の処理がブロックされる 「BEGIN してから COMMIT まで」の時間が長いほど、その間ロックが保持されて他のトランザクションが待機する。外部API・重い計算・ファイル操作などはトランザクションの外に出す。

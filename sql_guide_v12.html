はじめに — 代表的なデータベースの種類

「SQL」はデータベースを操作するための言語だが、DBの種類によって構文・関数・機能に差異がある。同じSQLでも動かないケースがあるため、どのDBを使っているかを常に意識する。

この教材はMySQL 8.0以降をベースに記載する。 差分はMySQL / PostgreSQL / SQLite比較で補足し、その他のDBは存在の紹介にとどめる。

はじめに — 基本用語

SQLを学ぶ前に、よく出てくる用語を押さえておく。

データをまとめて管理する入れ物。Excelのブック全体に相当する。

SQL

CREATE TABLE — テーブルを作成するSQL

実際にこのテーブルを作るSQL。本番環境では直接実行せず、マイグレーションツール経由が一般的。

-- データベースを作成して選択
CREATE DATABASE sample_db CHARACTER SET utf8mb4;
-- ↑ CHARACTER SET utf8mb4: 日本語などを扱う場合は必須。指定がないと文字化けエラーになる
USE sample_db;  -- MySQL固有。PostgreSQLは \c sample_db

-- departments（先に作る。usersがFKで参照するため）
CREATE TABLE departments (
    id   INT          PRIMARY KEY AUTO_INCREMENT,  -- 主キー。自動採番
    name VARCHAR(100) NOT NULL                      -- NULL禁止
) CHARACTER SET utf8mb4;

-- users
CREATE TABLE users (
    id            INT          PRIMARY KEY AUTO_INCREMENT,
    name          VARCHAR(100) NOT NULL,
    age           INT,
    department_id INT,
    created_at    DATETIME     DEFAULT CURRENT_TIMESTAMP,
    -- ↑ DATETIME型にはCURRENT_TIMESTAMPを使う。CURRENT_DATEはDATE型専用
    FOREIGN KEY (department_id) REFERENCES departments(id)  -- 外部キー制約
) CHARACTER SET utf8mb4;

-- orders
CREATE TABLE orders (
    id         INT           PRIMARY KEY AUTO_INCREMENT,
    user_id    INT           NOT NULL,
    amount     INT           NOT NULL,
    status     VARCHAR(50)  DEFAULT 'pending',
    ordered_at DATETIME      DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
) CHARACTER SET utf8mb4;
⚠ エラーになりやすいポイント2つ 1. DATETIME型にはCURRENT_TIMESTAMPを使う — MySQLのバージョンによって DATETIME DEFAULT CURRENT_DATE が使えないケースがある。DATETIME型には CURRENT_TIMESTAMP（または NOW()）を使う方が安全。DATE型専用の値をDATETIMEに指定したことが原因のエラーでもある。
2. 日本語を扱う場合はCHARACTER SET utf8mb4が必要 — 指定がないとデフォルトのlatin1になり、日本語INSERT時に Incorrect string value エラーが出る。データベース作成時かテーブル作成時に指定する。
AUTO_INCREMENT はMySQL固有。PostgreSQLでは SERIAL または GENERATED ALWAYS AS IDENTITY を使う。DB固有

SELECT — データを取得する

最も基本の構文。どのテーブルからどのカラムを取るかを指定する。

基本構文の形

SELECT   カラム名
FROM     テーブル名
WHERE    条件
ORDER BY 並び順
LIMIT    件数;

全カラム / カラムを絞る

-- * で全カラム（本番では必要なカラムだけ指定する方がパフォーマンスが良い）
SELECT * FROM users;

-- 必要なカラムだけ指定
SELECT
    id,
    name,
    age
FROM users;

DISTINCT — 重複を除いて取得する

-- 重複を除いてdepartment_idの一覧を取得
SELECT DISTINCT department_id FROM users;
-- usersに (1,1,2,NULL) があれば → 1, 2, NULL の3件が返る

-- 複数カラムの場合: 組み合わせが重複しているものを除く
SELECT DISTINCT department_id, age FROM users;
-- department_id と age の組み合わせが同じ行が除かれる
⚠ DISTINCTはパフォーマンスに注意 重複除去のために全件ソートが走る。大量データに多用すると遅くなる。「そもそもなぜ重複が発生しているか」を先に確認するのが望ましい（JOINのミスで重複が生じているケースが多い）。

NULLの演算と COALESCE

NULLは「値がない」という特殊な状態。NULLを含む計算は結果もNULLになる。これを知らないと集計や計算結果がおかしくなる。

-- NULLを含む演算はすべてNULLになる
-- NULL + 1    → NULL
-- NULL * 100  → NULL
-- NULL || 'さん' → NULL（文字列結合も）

-- COALESCE: NULLだったら代替値を返す
SELECT
    name,
    COALESCE(department_id, 0) AS dept_id,  -- NULLなら0
    COALESCE(age, 0) * 2         AS doubled   -- NULLなら0にしてから計算
FROM users;

-- 集計関数はNULLを無視する（COUNT(*)だけは例外）
SELECT
    COUNT(*)             AS total_rows,    -- 全行数（NULLも含む）
    COUNT(department_id) AS has_dept,      -- NULLを除いた件数
    AVG(age)             AS avg_age        -- NULLを除いた平均（NULLは分母に含まない）
FROM users;

LIKE のエスケープ

LIKE で使う % や _ 自体を文字として検索したいときはエスケープが必要。

-- 「10%」という文字列を含む行を検索したい場合
-- そのまま書くと % がワイルドカードとして解釈される
SELECT * FROM items
WHERE name LIKE '%10\%%' ESCAPE '';  -- \ でエスケープ

-- _ のエスケープ（任意の1文字ではなく _ そのものを検索）
SELECT * FROM items
WHERE name LIKE 'user\_name' ESCAPE '';
ESCAPEで使う文字は \ 以外でも指定できる。ESCAPE '!' とすれば ! をエスケープ文字にできる。DBによってデフォルトのエスケープ文字が違うので、明示的にESCAPE句を書く方が安全。

括弧で優先順位を変える

AND は OR より優先順位が高い。意図した通りに動かすには括弧で明示的にグループ化する。

-- 括弧なし: AND が先に評価される
-- 実際の解釈: (age >= 30 AND department_id = 1) OR name LIKE '鈴木%'
SELECT * FROM users
WHERE age >= 30 AND department_id = 1 OR name LIKE '鈴木%';

-- 括弧あり: 意図を明示する
-- 「30歳以上 かつ（開発部 または 鈴木さん）」
SELECT * FROM users
WHERE  age >= 30
  AND (department_id = 1 OR name LIKE '鈴木%');
AND と OR を混在させるときは常に括弧をつける習慣をつけると意図しない動作を防げる。

ORDER BY — 並び替え

キーワード意味例（age）
ASC昇順（小さい順・古い順）。省略するとデフォルトでASC23, 28, 32, 45
DESC降順（大きい順・新しい順）45, 32, 28, 23
-- 年齢の降順（大きい順）
SELECT * FROM users
ORDER BY age DESC;

-- 複数カラムで並び替え
-- department_idで昇順に並べて、同じ部署内ではageを降順
SELECT * FROM users
ORDER BY department_id ASC, age DESC;

NULLを後ろに持ってくる

MySQLではNULLはASC（昇順）のとき先頭に来る。NULLを最後に回したいときは2通りの書き方がある。

-- 方法1: 式（計算結果）で並び替え
-- IS NULL は NULL のとき1、値があるとき0 を返す
-- まず「NULLかどうか（0か1か）」で並べてからdepartment_idで並べる
SELECT * FROM users
ORDER BY department_id IS NULL ASC,  -- NULL=1が後ろ（非NULLの0が先）
         department_id ASC,
         age DESC;
-- ↑ 同じカラムを2回書いてもOK。役割が違う（1回目は「NULLか否か」の判定、2回目は値の順序）

-- 方法2: CASE式で明示的に書く（意図が読みやすい）
SELECT * FROM users
ORDER BY
    CASE WHEN department_id IS NULL THEN 1 ELSE 0 END ASC,  -- NULLは1（後回し）
    department_id ASC,
    age DESC;
ORDER BY句には式・関数・CASE式の結果を書ける。「計算した値で並び替える」という発想が持てると複雑な並び順も書けるようになる。両方とも結果は同じで、方法1は短く書けるが、方法2のCASE式版の方が何をしているか一目でわかる。チームで書く場合はCASE式推奨。

INSERT / UPDATE / DELETE — データを操作する（CRUD）

CRUD とは Create（作成）・Read（読取）・Update（更新）・Delete（削除）の頭文字。SELECT が Read にあたる。

ロールバックとトランザクション

UPDATE・DELETE などの変更操作は、やり直しがきかない場合がある。トランザクションで囲むと、問題があったときに変更をなかったことにできる（ロールバック）。

BEGIN;  -- トランザクション開始（START TRANSACTIONとも書く）

UPDATE users
SET   age = 33
WHERE id  = 1;

-- ここで SELECT して確認
SELECT * FROM users WHERE id = 1;

COMMIT;    -- 問題なければ確定（これで初めてDBに反映される）
-- ROLLBACK; -- 問題があればなかったことにする
TRUNCATEはトランザクションが効かないDBが多い。重要データには使わないこと。DB固有

論理削除 — データを残したまま「削除済み」にする

実務では DELETE でデータを物理的に消す代わりに、is_deleted のようなフラグカラムを使って「削除済み」とマークする設計が多い。これを論理削除という。

-- 物理削除（データが消える・取り消し不可）
DELETE FROM employees WHERE id = 5;

-- 論理削除（データは残る・フラグだけ変える）
UPDATE employees
SET   is_deleted = 1
WHERE id = 5;

-- 有効なレコードだけ取得するときは常に is_deleted = 0 を条件に加える
SELECT * FROM employees WHERE is_deleted = 0;

フラグカラムの読み方

プログラミングでは true（真）= 1、false（偽）= 0 という対応が基本。DBのフラグカラムもこれに基づいていて、TINYINT(1)型の 0/1 がそのまま false/true に対応している。

DB格納値プログラムでの意味状態のイメージ
0false（偽）フラグが立っていない・条件を満たしていない
1true（真）フラグが立っている・条件を満たしている

これを踏まえると is_deleted = 0 は「削除フラグが false」＝「削除されていない」、is_deleted = 1 は「削除フラグが true」＝「削除済み」と読める。

MySQLでは TINYINT(1) が事実上の BOOLEAN 型として使われている。PostgreSQL は BOOLEAN 型が独立して存在し、TRUE/FALSE または 1/0 どちらでも書ける。DB固有

-- MySQL: TINYINT(1)で0/1を使う
WHERE is_deleted = 0   -- false = 削除されていない
WHERE is_deleted = 1   -- true  = 削除済み

-- PostgreSQL: BOOLEAN型ならTRUE/FALSEも使える
WHERE is_deleted = FALSE
WHERE is_deleted = TRUE

カラム名によって「0が何を表すか」が変わるので、名前をよく確認する習慣をつける。

よく見るカラム名0の意味1の意味備考
is_deleted有効（削除されていない）削除済み論理削除の定番。最もよく見る
is_active無効（アクティブでない）有効・アクティブis_deletedと向きが逆になるので注意
is_published非公開公開済み記事・商品の公開フラグ

⚠ is_active は is_deleted と向きが逆 is_deleted = 0 が有効なのに対し、is_active = 1 が有効。混在しているシステムでは条件を逆に書くミスが起きやすい。カラム名を毎回確認する。
論理削除を使う理由：外部キーで参照されているデータを物理削除するとFK制約エラーになる。また、削除後も履歴として残す必要がある場合（注文履歴・退職者の情報等）に論理削除が有効。

DROP TABLE — テーブルごと削除する

-- テーブルを削除する（データも定義も全て消える・取り消し不可）
DROP TABLE users;

-- テーブルが存在する場合のみ削除（存在しないとエラーになるのを防ぐ）
DROP TABLE IF EXISTS users;

-- 複数テーブルを一度に削除
DROP TABLE IF EXISTS orders, users, departments;
-- ↑ FK制約がある場合は参照している側（子）から先に削除する
⚠ DROP TABLEはテーブルの定義ごと消える DELETEやTRUNCATEはデータだけ消すがテーブルの器は残る。DROP TABLEはテーブル自体が消えて取り消せない。必ずバックアップを確認してから実行する。

SELECT の結果をそのまま INSERT できる。テーブルのコピーや集計結果の保存に使う。

ALTER TABLE — テーブルの構造を変更する

テーブルを作成した後でカラムの追加・変更・削除やインデックスの追加を行う。本番環境では影響範囲を確認してから実行する。

-- カラムを追加する
ALTER TABLE users
ADD COLUMN email VARCHAR(255);

-- カラムを追加する（位置を指定）
ALTER TABLE users
ADD COLUMN email VARCHAR(255) AFTER name;  -- nameの直後に追加

-- カラムの型・制約を変更する
ALTER TABLE users
MODIFY COLUMN email VARCHAR(512) NOT NULL;  -- 型変更＋NOT NULL追加

-- カラム名と型を同時に変更する
ALTER TABLE users
CHANGE COLUMN email mail_address VARCHAR(255);  -- email → mail_address

-- カラムを削除する
ALTER TABLE users
DROP COLUMN email;

-- インデックスを追加する
ALTER TABLE users
ADD INDEX idx_email (email);

-- 外部キー制約を追加する
ALTER TABLE users
ADD FOREIGN KEY (department_id) REFERENCES departments(id);

-- テーブル名を変更する
ALTER TABLE users RENAME TO app_users;
⚠ 本番環境のALTER TABLEは慎重に 大きいテーブルへのALTER TABLEはテーブルロックが発生して他のクエリが止まる場合がある（MySQLバージョン・操作内容による）。本番実行前に必ずバックアップを取り、影響範囲・実行時間を確認する。カラム削除は取り消せないので特に注意。

CASE式 — SQLの条件分岐

CASE式はSQLの中で条件分岐を書く構文。SELECT・WHERE・ORDER BY・GROUP BYのどこでも使える。

基本の書き方（検索CASE式）

-- CASE WHEN 条件 THEN 結果 ... ELSE デフォルト END
SELECT
    name,
    age,
    CASE
        WHEN age < 20 THEN '未成年'
        WHEN age < 65 THEN '成人'
        ELSE               'シニア'   -- どの条件にも合わない場合
    END AS age_group
FROM users;

単純CASE式 — 1つのカラムの値で分岐

SELECT
    name,
    CASE department_id        -- カラム名を先に書く
        WHEN 1 THEN '開発部'
        WHEN 2 THEN '営業部'
        WHEN 3 THEN '総務部'
        ELSE       '未所属'
    END AS dept_name
FROM users;

CASE式はORDER BY・GROUP BY・WHERE でも使える

-- ORDER BY: カスタム順序での並び替え（NULLを後ろに、など）
SELECT * FROM users
ORDER BY
    CASE WHEN department_id IS NULL THEN 1 ELSE 0 END ASC,
    department_id ASC;

-- GROUP BY: 集計のグループをCASEで作る
SELECT
    CASE
        WHEN age < 30 THEN '30歳未満'
        ELSE               '30歳以上'
    END AS age_group,
    COUNT(*) AS cnt
FROM     users
GROUP BY
    CASE
        WHEN age < 30 THEN '30歳未満'
        ELSE               '30歳以上'
    END;

括弧で1行にする書き方

単純な条件分岐なら括弧でまとめてSELECT句やORDER BY句を1行で書ける。

-- ORDER BY での1行CASE（NULLを後ろに）
SELECT * FROM users
ORDER BY
    (CASE WHEN department_id IS NULL THEN 1 ELSE 0 END) ASC,
    department_id ASC,
    age DESC;

-- SELECT句での1行CASE（フラグを日本語ラベルに変換）
SELECT
    name,
    (CASE WHEN department_id IS NULL THEN '未所属' ELSE '所属あり' END) AS dept_status
FROM users;
WHEN～ENDが1パターンだけなら括弧でまとめて1行にするとすっきり読める。WEHNが複数になる場合は複数行に分けた方が見やすい。
⚠ ELSEを省略するとNULLになる ELSEを書かないと、どの条件にも当てはまらなかった場合の結果がNULLになる。意図していない場合は必ずELSEを書く。

GROUP BY / HAVING / 集計関数

データをグループ化して集計する。「部署ごとの人数」「ユーザーごとの合計金額」などを出すときに使う。

集計関数一覧

関数内容例
COUNT(*)行数を数える（NULLも含む）COUNT(*) → 4
COUNT(col)NULLを除いた行数COUNT(department_id) → 3
SUM(col)合計値SUM(amount) → 13000

GROUP BY — グループごとに集計

-- 部署ごとの人数と平均年齢
SELECT
    department_id,
    COUNT(*) AS user_count,
    AVG(age) AS avg_age
FROM     users
GROUP BY department_id
ORDER BY user_count DESC;

HAVING — グループに対して条件を絞る

-- 人数が2人以上の部署だけ
SELECT
    department_id,
    COUNT(*) AS user_count
FROM     users
GROUP BY department_id
HAVING   COUNT(*) >= 2;
⚠ WHERE と HAVING の違い WHERE : グループ化「前」に行を絞る。集計関数（COUNT等）は使えない
HAVING : グループ化「後」に絞る。集計関数が使える

NG: WHERE COUNT(*) >= 2 ← 集計関数はWHEREに書けない
OK: HAVING COUNT(*) >= 2

ROLLUP — 小計・合計を一緒に出す

GROUP BYに WITH ROLLUP をつけると、グループごとの集計に加えて全体の合計行が自動で追加される。MySQL固有 / PostgreSQLはROLLUP()構文

-- 部署ごとの人数 + 全体合計を1つのSELECTで出す
SELECT
    department_id,
    COUNT(*) AS user_count
FROM     users
GROUP BY department_id WITH ROLLUP;
-- 結果イメージ:
-- department_id | user_count
-- 1             | 2          ← 部署1のグループ（山田・鈴木）
-- 2             | 1          ← 部署2のグループ（田中）
-- NULL          | 1          ← 部署NULLのグループ（佐藤）
-- NULL          | 4          ← ROLLUPが追加した全体合計行
--
-- ⚠ 部署NULLのグループ行とROLLUPの合計行が両方NULLになるため区別できない。
--   COALESCEをかけると佐藤さんの行も「全体合計」になってしまう（NG）。
--   GROUPING()関数を使うとROLLUPが追加した行かどうかを区別できる。

-- NG: COALESCEだけだと佐藤さんの行も「全体合計」になってしまう
SELECT
    COALESCE(CAST(department_id AS CHAR), '全体合計') AS dept,
    COUNT(*) AS user_count
FROM     users
GROUP BY department_id WITH ROLLUP;
-- 結果: 1=2件, 2=1件, 全体合計=1件（佐藤）, 全体合計=4件（ROLLUP）← 区別できていない

-- OK: GROUPING()でROLLUPの合計行かどうかを判定する
-- GROUPING(col)はROLLUPが追加した行のとき1、通常のグループ行のとき0を返す
SELECT
    CASE
        WHEN GROUPING(department_id) = 1 THEN '全体合計'   -- ROLLUPの合計行
        WHEN department_id IS NULL          THEN '未所属'    -- 部署NULLのグループ（佐藤）
        ELSE CAST(department_id AS CHAR)
    END AS dept,
    COUNT(*) AS user_count
FROM     users
GROUP BY department_id WITH ROLLUP;
-- 結果: 1=2件, 2=1件, 未所属=1件（佐藤）, 全体合計=4件 ← 正しく区別できる

LEFT JOINで「どちらを左に置くか」の考え方

「全件漏らさず出したいテーブルを左に置く」というのが基本の判断軸。LEFT JOINは左のテーブルを必ず全行出して、右のテーブルは一致した行だけ付加するという動きをする。

言い換えると、「主役のテーブル」が左、「付加情報を取ってくるテーブル」が右。

-- 例1: 注文を全件出して、各注文のユーザー名を付加したい
-- → 主役は orders（全件出したい）、付加情報は users のname
SELECT o.id, o.amount, u.name
FROM      orders o          -- 主役（全件出す）が左
LEFT JOIN users  u ON o.user_id = u.id;
-- user_idがNULLの注文もNULLのまま出てくる

-- 例2: ユーザーを全員出して、各ユーザーの注文を付加したい
-- → 主役は users（全員出したい）、付加情報は orders
SELECT u.name, o.amount
FROM      users  u          -- 主役（全員出す）が左
LEFT JOIN orders o ON u.id = o.user_id;
-- 注文がないユーザーはamountがNULLで出てくる
「行数が多い方を左に」という感覚は結果的に合っていることが多い。ただ本質は行数ではなく「どちらを全件出したいか」。行数が少なくても全員出したければそちらが左になる。迷ったら「このSELECTの主役は何か」を先に決めると判断しやすい。

JOIN後は件数を確認する

JOINすると元のテーブルより行数が増えることがある。「ユーザー一覧に部署情報を紐づけたいだけなのに行が増えた」というのはJOINミスの典型パターン。JOINの前後でCOUNTを比べる習慣をつけると気づきやすい。

-- 想定シナリオ:
-- mastersテーブルに同じcodeが複数行存在していた場合

-- まずJOIN前の件数を確認（期待値）
SELECT COUNT(*) FROM transactions;          -- 例: 1000件

-- JOINしてみる
SELECT COUNT(*)
FROM       transactions t
INNER JOIN masters      m ON t.code = m.code;  -- 3000件になった → 増えてる！

-- 原因を調べる: mastersにcodeの重複があるか確認
SELECT code, COUNT(*) AS cnt
FROM     masters
GROUP BY code
HAVING   COUNT(*) > 1;  -- 重複しているcodeが見つかる
⚠ JOINで行が増えるのはマスタ側に重複があるサイン LEFT JOINで「ユーザー1人に対して部署が複数紐づいていた」場合も同様に行が増える。JOIN後に意図しない件数になっていたら、まず結合キーの重複を疑う。
-- 対処パターン1: DISTINCT で重複を排除（一時的な応急処置）
SELECT DISTINCT t.*
FROM       transactions t
INNER JOIN masters      m ON t.code = m.code;

-- 対処パターン2: サブクエリでマスタ側を先に1件に絞る（根本対処）
SELECT t.*
FROM       transactions t
INNER JOIN (
    SELECT DISTINCT code, name  -- 重複を除いた1件のマスタを作る
    FROM masters
) m ON t.code = m.code;

エイリアス（AS）

テーブル名やカラム名に一時的な別名をつける機能。JOINでテーブル名が長くなるときに特に役立つ。

カラムのエイリアス — 結果の列名を変える

SELECT
    name        AS ユーザー名,   -- 結果の列名が「ユーザー名」になる
    age         AS 年齢,
    COUNT(*)    AS 件数          -- 集計結果にも名前をつけられる
FROM users;

テーブルのエイリアス — JOINで必須レベルで使う

-- u = users の略、d = departments の略
SELECT
    u.name,          -- どのテーブルのnameか明示できる
    d.name AS dept
FROM       users       u  -- AS は省略可能
INNER JOIN departments d ON u.department_id = d.id;
AS は省略できる。FROM users u と書くだけでエイリアスになる。複数テーブルを使うときは u.name のようにテーブル名（エイリアス）を先頭につけてどちらのテーブルのカラムか明示する。

クエリの中に別のクエリを埋め込む。集計結果や別テーブルの値を条件に使いたいときに使う。

WHERE 句のサブクエリ

-- 平均年齢より上のユーザーを取得
SELECT * FROM users
WHERE age > (
    SELECT AVG(age) FROM users
);

FROM 句のサブクエリ（派生テーブル）

-- 部署ごとの集計結果をさらに絞る
SELECT * FROM (
    SELECT
        department_id,
        COUNT(*) AS cnt
    FROM     users
    GROUP BY department_id
) AS dept_summary       -- サブクエリにもエイリアスが必要
WHERE cnt >= 2;

EXISTS — サブクエリで存在確認

-- 注文が1件でも存在するユーザーだけ取得
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o  -- 1は存在確認だけなので何でもいい
    WHERE o.user_id = u.id
);
EXISTS は IN より高速なことが多い。大量データを扱う場合は EXISTS を優先する。

SELECT 句のサブクエリ（スカラサブクエリ）

SELECT句の取得列としてサブクエリを書く。必ず1行1列（スカラ値）を返すサブクエリでなければならない。外側のクエリの各行に対して1回ずつ実行される。

-- 各ユーザーの注文合計金額を列として付加する
SELECT
    u.id,
    u.name,
    (SELECT SUM(amount)
     FROM  orders o
     WHERE o.user_id = u.id   -- 外側のu.idを参照（相関サブクエリ）
    ) AS total_amount
FROM users u;

-- 各ユーザーの部署名を列として付加する
SELECT
    u.name,
    (SELECT d.name
     FROM  departments d
     WHERE d.id = u.department_id
    ) AS dept_name   -- JOINの代わりに列として取得するパターン
FROM users u;
⚠ スカラサブクエリは2行以上返すとエラーになる サブクエリが複数行を返す場合（結合キーに重複がある場合など）は実行時エラーになる。必ず1行1列になることを確認してから使う。
また外側の行数分だけサブクエリが実行されるため、大量データでは遅くなりやすい。JOINで書き直せる場合はJOINの方が速いことが多い。
-- 上記の「部署名を付加する」クエリをJOINで書き直した場合
-- 動作は同じだがこちらの方が一般的に速い
SELECT
    u.name,
    d.name AS dept_name
FROM      users       u
LEFT JOIN departments d ON u.department_id = d.id;

UNION / UNION ALL — 複数のSELECT結果を縦に結合する

2つ以上のSELECT結果を縦につなげて1つの結果にする。JOINが「横に結合」するのに対し、UNIONは「縦に結合」するイメージ。

JOINとUNIONの違い

JOINは「同じ行に別テーブルの情報を付け加える」（カラムが増える）。UNIONは「別々の検索結果を1つにまとめる」（行が増える）。目的が全く違う。

UNION — 重複を除いて結合

-- 開発部のユーザーと、30歳以上のユーザーを合わせて取得（重複は除く）
SELECT id, name FROM users WHERE department_id = 1
UNION
SELECT id, name FROM users WHERE age >= 30;
-- 両方の条件に当てはまる人は1件だけ出てくる

UNION ALL — 重複を除かずに結合

-- 重複をそのまま残して結合（DISTINCTの処理がない分速い）
SELECT id, name FROM users WHERE department_id = 1
UNION ALL
SELECT id, name FROM users WHERE age >= 30;
-- 両方の条件に当てはまる人は2件出てくる（重複あり）
UNIONUNION ALL
重複除く（DISTINCTと同じ処理が走る）除かない
速度遅い（重複除去のためにソートが走る）速い
使いどころ重複が出ることがわかっていて除きたい場合重複が出ないことがわかっている場合・意図的に全件欲しい場合
⚠ UNIONの制約 ① 全てのSELECTのカラム数と型が一致していなければならない
② ORDER BYは最後のSELECTにだけ書く（全体の並び替えになる）
③ 重複が出ないことがわかっている場合はUNION ALLを使う（UNIONは不要なソートが走る）
-- 実務でよくあるパターン: 異なるステータスを1つにまとめる
SELECT 'completed' AS status, COUNT(*) AS cnt FROM orders WHERE status = 'completed'
UNION ALL
SELECT 'pending',               COUNT(*) AS cnt FROM orders WHERE status = 'pending'
UNION ALL
SELECT 'cancelled',             COUNT(*) AS cnt FROM orders WHERE status = 'cancelled'
ORDER BY cnt DESC;  -- ORDER BYは最後だけに書く

INTERSECT / EXCEPT — 共通部分・差分を取る

⚠ MySQL 8.0.31以降でないと使えない INTERSECTとEXCEPTはMySQL 8.0.31以降で追加された機能。それ以前のバージョンでは使えない。PostgreSQL・SQLiteは古いバージョンから対応している。本番環境のMySQLバージョンを必ず確認すること。DB固有
-- INTERSECT: 両方のSELECT結果に共通する行だけを取得（AND的な発想）
-- 「開発部に所属している」かつ「30歳以上」のユーザー
SELECT id, name FROM users WHERE department_id = 1
INTERSECT
SELECT id, name FROM users WHERE age >= 30;

-- EXCEPT: 左のSELECT結果から右のSELECT結果を引いた行を取得（差集合）

EXCEPT
SELECT id, name FROM users WHERE age >= 30;

アップサート — あれば更新、なければ挿入

同じキー（主キーや一意キー）のデータがすでに存在すれば UPDATE、なければ INSERT を1文で行う。DBによって構文が異なる。

MySQL MySQL固有

INSERT INTO users (id, name, age)
VALUES (1, '山田 太郎', 33)
-- ↑ まずINSERTを試みる。id=1 がすでに存在すれば重複エラーが発生する
ON DUPLICATE KEY UPDATE
-- ↑ 重複エラーが発生した場合にここのUPDATEを実行する（エラーにはならない）
    name = VALUES(name),  -- VALUES(カラム名) = 「今INSERTしようとした値」を参照する
    age  = VALUES(age);   -- つまり VALUES(1, '山田 太郎', 33) の中の値のこと
VALUES(col) の書き方はMySQL 8.0.20以降では非推奨。代わりに name = alias.name の形（行エイリアス）を使うことが推奨されている。MySQL固有

PostgreSQL PostgreSQL固有

INSERT INTO users (id, name, age)
VALUES (1, '山田 太郎', 33)
-- ↑ まずINSERTを試みる
ON CONFLICT (id)
-- ↑ id カラムで重複が起きた場合（主キー重複など）
DO UPDATE SET
-- ↑ 重複したら以下のUPDATEを実行する
    name = EXCLUDED.name,
--        ↑ EXCLUDED = 「今INSERTしようとした値」を指す特別なキーワード
--          つまり VALUES で指定した '山田 太郎' のこと
    age  = EXCLUDED.age;  -- 同様に VALUES の 33 のこと
MySQLは ON DUPLICATE KEY UPDATE、PostgreSQLは ON CONFLICT ... DO UPDATE SET と書き方が違う。どちらも「重複したら更新する」という意味は同じ。

ウィンドウ関数

GROUP BYと似ているが「行を集約せずに」集計値を各行に付加できる。ランキングや累計計算で使う。

GROUP BYウィンドウ関数
結果行数グループ数に減る元の行数のまま
用途集計サマリ各行に集計値を付加
構文GROUP BY colOVER (PARTITION BY col)

基本構文

関数名() OVER (
    PARTITION BY グループ化するカラム  -- GROUP BY相当（省略可）
    ORDER BY     並び順               -- 省略可

ROW_NUMBER / RANK / DENSE_RANK — ランキング

-- 部署ごとに年齢の高い順でランキング
SELECT
    name,
    department_id,
    age,
    ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY age DESC) AS row_num,
    RANK()        OVER (PARTITION BY department_id ORDER BY age DESC) AS rank_num,
    DENSE_RANK() OVER (PARTITION BY department_id ORDER BY age DESC) AS dense_rank_num
FROM users;
関数同順位があるとき例（同率2位が2人）
ROW_NUMBER同順位でも連番1, 2, 3, 4
RANK同順位は同じ番号、次をスキップ1, 2, 2, 4
DENSE_RANK同順位は同じ番号、スキップなし1, 2, 2, 3

SUM — 累計

-- ユーザーごとの累計注文金額
SELECT
    user_id,
    ordered_at,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id    -- ユーザーごとに区切る
        ORDER BY     ordered_at  -- 注文日順に累計する
    ) AS cumulative_amount
FROM orders;

LAG / LEAD — 前後の行の値を取得

-- 前の注文金額と比較（LAG: 前の行 / LEAD: 次の行）
SELECT
    user_id,
    ordered_at,
    amount,
    LAG(amount, 1) OVER (
        PARTITION BY user_id
        ORDER BY     ordered_at
    ) AS prev_amount  -- 1つ前の注文金額（先頭行はNULL）
FROM orders;
ウィンドウ関数は SELECT 句にしか書けない。WHERE や HAVING には書けないので、条件で使いたい場合はサブクエリで包む。

Part 1 — 基礎

🔷01 TypeScriptとはなぜ型が必要か

TypeScript（TS）はMicrosoftが開発した、JavaScriptに静的型付けを追加した言語。書いたコードはブラウザ実行前にJavaScriptへ変換（トランスパイル）される。

JSとTSの最大の違い

❌ JavaScript — 実行するまでエラーに気づかない
function double(n) {
  return n * 2
}
double('hello')  // NaN になるが書いた時点では気づかない
✅ TypeScript — 書いた瞬間にエラーがわかる
function double(n: number): number {
  return n * 2
}
double('hello')
// エラー: string は number に代入できない

TypeScriptでできること

  • 🔍
    型チェック
    コンパイル時エラー検出
    実行前にバグを発見。VSCodeがリアルタイムでエラーを教えてくれる。「実行してみたら壊れた」がなくなる
  • ⚡
    補完・リファクタ
    IDE支援が爆発的に強くなる
    プロパティ名の補完・関数の引数ヒント・リネームが安全にできる。タイポが即座にわかる
  • 📖
    ドキュメント化
    型がそのままドキュメントになる
    関数の型定義を見るだけで「何を渡して何が返るか」がわかる。コメントを書かなくていい
JavaScriptはそのままTypeScriptとして動く。既存のJSファイルを.tsに変えるだけで始められる。
🧱02 基本の型プリミティブ・any・unknown・never

TypeScriptのすべての値はいずれかの「型」に属する。まず最も基本的なプリミティブ型（値そのもの）を把握しよう。

プリミティブ型一覧

string
'山田' / "hello" / `${name}`
文字列。シングル・ダブルクォート・テンプレートリテラル（バッククォート）すべて同じ型
number
32 / 1980.5 / 0xff
整数・小数・16進数すべてがnumber一種類。PHPのint/floatの区別はない
boolean
true / false
true/falseのみ。0・null・""はbooleanではないので型エラーになる
null
null
「意図的に値がない」状態。APIが「データなし」を返すときなど、明示的にセットする値
undefined
undefined
「まだ値がない・セットされていない」状態。オプショナルなプロパティのデフォルト値
bigint
9007199254740992n
numberの安全な上限（253-1）を超える整数。末尾にnを付ける。IDや金額の大きな数値に使う
symbol
Symbol('key')
絶対にかぶらない一意の値を作る。Symbol('key') === Symbol('key') は false（同じ文字列を渡しても別物）。オブジェクトのキーや定数の識別子に使う。実務での出現頻度は低いが見かけたら慌てなくていい
// 型注釈の書き方（: のあとに型名）
const name:    string    = '山田太郎'
const age:     number    = 32
const active:  boolean   = true
const score:   number    = 98.5    // 小数もnumber（PHPのfloatに相当）

// null と undefined は別物
let selected: string | null      = null       // APIから「選択なし」が返ってくる想定
let loaded:   string | undefined = undefined  // まだフェッチしていない状態

number の落とし穴 — 精度に注意

JavaScriptのnumberはIEEE 754の浮動小数点数。整数の計算では基本的に問題ないが、小数の演算は誤差が出る。金額・税率の計算をそのままnumberでやると壊れることがある。

❌ 小数の加算でずれる
// これはJavaScript/TypeScript共通の問題
0.1 + 0.2  // → 0.30000000000000004
✅ 整数で扱うか、専用ライブラリを使う
// 金額は「円」単位（整数）で持つ
const price: number = 1980  // 円（整数）
// または decimal.js などを使う

null vs undefined — 使い分けの基準

TypeScriptではnullとundefinedは別の型として扱われる（strictNullChecks: true時）。どちらを使うかはプロジェクトの慣習による部分もあるが、意味に違いがある。

値意味使いどころ
null意図的に「値なし」をセットAPIレスポンスの「データなし」・初期化後に空にした状態
undefinedまだ値が設定されていないオプショナルプロパティ（?）のデフォルト・まだフェッチしていない状態
// null: 意図的に「値なし」を入れる
let selectedUser: User | null = null   // まだ選択していない
selectedUser = fetchUser(1)              // 選択後: Userオブジェクト
selectedUser = null                      // 選択を解除 → 意図的なnull

// undefined: まだ値が設定されていない（省略されたプロパティのデフォルト）
interface Profile { age?: number }    // age は省略可（optional）
const p: Profile = {}
console.log(p.age)  // undefined — エラーではなく「まだ値がない」状態

特殊な型（重要）

any
型チェックを無効化
何でも入る。TypeScriptの型チェックを完全に無効にする。基本的に使わない。使う場面はJSからTSへの移行中など「どうしても型が付けられない一時的な状況」のみ。型がわからないときはunknownを使う
unknown
安全なany
何でも受け取れるが、使う前に型チェックが必須。外部からの入力・JSONのparse結果など「型が保証できない値」に使う
never
絶対に到達しない
例外のみスローする関数の戻り値・Union型の網羅チェックのケース漏れ検出に使う。「ここには絶対来ない」という証明
void
戻り値なし
console.logのような「値を返さない関数」の戻り値型。関数の引数で「戻り値を使わないコールバック」にも使う
// never の使いどころ — Union型の網羅チェック
type Status = 'active' | 'inactive' | 'banned'

function handle(s: Status) {
  switch (s) {
    case 'active':   return 'アクティブ'
    case 'inactive': return '非アクティブ'
    case 'banned':   return 'BANされた'
    default:
      const _: never = s  // Status に新しいケースを追加したらここでエラー
  }
}
❌ any — 型チェックが死ぬ
let value: any = 'hello'
value = 42        // OK（何でも代入できる）
value.toFixed()   // 実行時エラーになっても気づかない
✅ unknown — 使う前にチェックが強制される
let value: unknown = 'hello'
if (typeof value === 'string') {
  value.toUpperCase() // OK（string確定後）
}
⚠️ anyを書いたらTypeScriptの意味がない「型わからん → any」ではなく「型わからん → unknown で受けて絞り込む」が正解。anyはどうしても型が付けられない移行期のコードにのみ使い、コメントで理由を書くこと。

用語メモ：tsconfig.json の strict: true を設定すると noImplicitAny が有効になり、型が暗黙的に any になるコードはコンパイルエラーになる（明示的に : any と書けば通る）。新規プロジェクトでは strict: true が推奨。
🔍03 型推論TypeScriptが自動で型を決める仕組み

TypeScriptは多くの場合、初期値から型を自動で判定する。毎回書かなくていい。

// 書かなくても型が決まる（推論される）
const name = '山田'      // 推論: string
const age  = 32         // 推論: number
const flag = true       // 推論: boolean
const nums = [1, 2, 3]  // 推論: number[]

// ⚠️ const vs let — リテラル型の違い（重要）
const dir1 = 'left'  // 推論: "left"（リテラル型 — const は変更不可なので具体値で推論）
let   dir2 = 'left'  // 推論: string（let は別の値に変わりうるので広い型）
// → as const・型ガードを理解するうえで必須の挙動。S07 で詳しく扱う

// 関数の戻り値も推論される
function add(a: number, b: number) {
  return a + b  // 戻り値型は自動でnumberになる
}

型を明示すべき場面 vs 省略できる場面

場面判断理由
変数の初期化（値あり）✅ 省略OK値から推論される
関数の引数📝 明示する呼び出し側が間違えたとき検出できる
関数の戻り値📝 明示推奨実装ミスを防げる。ドキュメントになる
値なしで宣言した変数📝 明示するstrict: true ではコンパイルエラー（noImplicitAny違反）
複雑な型が推論される場所📝 明示する読みやすさのため
// 値なし宣言は明示必須
let name: string        // OK
let name2               // strict: true ではコンパイルエラー（noImplicitAny違反）

// 引数と戻り値は書く
function getUser(id: number): User {
  return users.find(u => u.id === id)!
}
「型を書きすぎ」より「型を書かなさすぎ」の方が危険。迷ったら書く。
⚠️04 null / undefined の扱いstrictNullChecks・?. ・?? ・!

TypeScriptのstrictモードではnull/undefinedは別の型として扱われる。JSで最も多いバグ「Cannot read property of null」をコンパイル時に防げる。

// strictNullChecks が有効な場合
let name: string = '山田'
name = null   // ❌ エラー: null は string に代入できない

// nullable にするには Union を使う
let name: string | null = '山田'
name = null   // ✅ OK

null/undefinedを扱う演算子 （?. ?? || はJavaScriptの機能・TypeScriptでもそのまま使える）

  • 🔗
    ?. オプショナルチェーン
    user?.address?.city
    nullまたはundefinedならundefinedを返す。エラーにならない。プロパティアクセス・メソッド呼び出し・配列インデックスすべてに使える
  • 🔀
    ?? Nullish合体演算子
    value ?? 'デフォルト'
    左辺がnull/undefinedのときだけ右辺を使う。0や''は「値がある」として通す。||との最大の違いはここ
  • 🔀
    || OR演算子（falsy全般でフォールバック）
    value || 'デフォルト'
    左辺が falsy（null / undefined / 0 / '' / false）のとき右辺を使う。?? との違いに注意：0 や '' もフォールバックされてしまう。「値があっても falsy なら置き換えたい」場面で使う
  • ⚠️
    ! Non-nullアサーション（最終手段）
    element!
    「絶対nullじゃない」と型チェックに宣言。実行時チェックなしで危険。if文による型ガードに置き換えるのが基本。どうしても必要な場合のみ使う
interface Address { city: string }
interface User { name: string; address?: Address }

const user: User | null = getUser(1)

// ?. — nullなら undefined になる（エラーにならない）
const city = user?.address?.city     // string | undefined

// ?? — null/undefinedのときのデフォルト値
const display = user?.name ?? '匿名'  // string

// || との違い: ?? は 0 や '' を通す
const count = data.count ?? 0  // countが0でも0を使う（|| だと 0 を飛ばす）

// ! の代替：優先順位は「型ガード（if文）> as > !」— as も実行時保証なし

// ✅ 基本：型ガード（if文）で絞り込む
const el = document.getElementById('app')
if (el instanceof HTMLDivElement) {
  el.style.color = 'red'  // この中では HTMLDivElement 型が保証される
}

// ⚠️ やむを得ない場合のみ：as（実行時保証なし）
const el2 = document.getElementById('app') as HTMLDivElement
const el2 = document.getElementById('app')!                   // △ 動くが理由が不明瞭
❌ || を使うと 0 や '' が消える
const count = data.count || 0
// data.count が 0 のとき → 0（デフォルト）になる
// フォームの入力件数などで誤動作する
✅ ?? なら 0 や '' を正しく扱える
const count = data.count ?? 0
// data.count が 0 → 0（本物の0）を使う
// null/undefined のときだけ 0 を使う

Part 2 — 型の構造

🏗️05 interface vs typeオブジェクト型の定義方法と使い分け

オブジェクトの「形」を定義する方法が2つある。どちらでも動くが、使い分けに意味がある。

// interface — オブジェクトの形を定義
interface User {
  id:      number
  name:    string
  email?:  string   // ? がついたプロパティは省略可能（optional）
  readonly createdAt: Date  // readonly = 再代入不可
}

// type alias — 同じことができる
type User = {
  id:      number
  name:    string
  email?:  string
  readonly createdAt: Date
}

決定的な違い

違いinterfacetype
Union型の定義
（型を「どちらか」で組み合わせる — 詳細はS06）		❌ できない✅ type ID = string | number
拡張extends で継承& で合成
同名での再定義✅ 合成される（Declaration Merging）❌ エラーになる
プリミティブ型に別名をつける❌ できない✅ type ID = string
// 拡張の書き方
interface AdminUser extends User {  // interface は extends
  role: 'admin'
  permissions: string[]
}

type AdminUser = User & {          // type は & (Intersection)
  role: 'admin'
  permissions: string[]
}

// type でしかできないこと
type ID = string | number
type Status = 'active' | 'inactive'

// Declaration Merging（ライブラリ型拡張に使う）
interface Window {
  myAppVersion: string  // 既存のWindow型にプロパティを追加
}

実務での使い分け方針

  • 🏷️
    interface を使う場面
    オブジェクト形状の定義
    User・Product・ApiResponseなど「モノの形」を定義するとき。extendsで継承できるので型の再利用がしやすい。また同名interfaceを複数書くとマージされるため、ライブラリの型拡張にも使われる
  • 🔧
    type を使う場面
    Union・プリミティブ・Mapped Types
    Union型（string | number）・リテラル型・ユーティリティ型（Partial<T>など）を使うとき。interfaceでは書けない複雑な型はtypeを使う
  • 📋
    実務の現実
    どちらでもいいケースが多い — チーム規約に従う
    多くのプロジェクトでは「オブジェクト→interface、それ以外→type」か「全部type」かどちらかに統一している。混在させないことが最も重要
迷ったら「オブジェクト形状 → interface、それ以外 → type」で始めると整理しやすい。
🔀06 Union / Intersection / リテラル型型を組み合わせる核心

Union型（|）— AまたはB

type StringOrNumber = string | number
type Status = 'active' | 'inactive' | 'pending'
type Nullable<T> = T | null

// Union型の配列は2通りの書き方がある（どちらも同じ意味）
const ids: (string | number)[] = [1, 'abc', 2]   // 配列型記法
const ids: Array<string | number> = [1, 'abc', 2]  // ジェネリクス記法（Union型を含む場合に読みやすい）

Intersection型（&）— AかつB

type Timestamps = {
  createdAt: Date
  updatedAt: Date
}
type UserWithTimestamps = User & Timestamps
// User の全プロパティ + Timestamps の全プロパティ を持つ

リテラル型 — 具体的な値で型を作る

type Direction = 'north' | 'south' | 'east' | 'west'
type HTTPStatus = 200 | 201 | 400 | 401 | 404 | 500
type Toggle = true | false  // boolean と同じだが意図が明確

function move(dir: Direction): void {
  // 'north'/'south'/'east'/'west' 以外は渡せない
}

Discriminated Union — 実務で最重要パターン

「kind」や「type」のような識別子フィールド（リテラル型の固定値）を持つUnion型。TypeScriptがその値を見て型を自動で絞り込んでくれる。

// 基本形：各パターンに kind フィールド（リテラル型）をつける
type Success = { kind: 'success'; data: string }
type Failure = { kind: 'failure'; error: string }
type Result = Success | Failure

// kind を見るだけで型が絞り込まれる
function handle(r: Result): void {
  if (r.kind === 'success') {
    console.log(r.data)   // ここは Success 型に確定
  } else {
    console.log(r.error)  // ここは Failure 型に確定
  }
}

パターンが増えても switch で網羅的に処理できる。

type Shape =
  | { kind: 'circle';    radius: number }
  | { kind: 'rectangle'; width: number; height: number }
  | { kind: 'triangle';  base: number;  height: number }

function area(shape: Shape): number {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2    // shape は circle 型に絞られる
    case 'rectangle':
      return shape.width * shape.height
    case 'triangle':
      return (shape.base * shape.height) / 2
  }
}

// APIレスポンスでよく使うパターン
type ApiResult<T> =
  | { ok: true;  data: T }
  | { ok: false; error: string }
Discriminated Unionは「状態管理」「APIレスポンス」「エラーハンドリング」どこでも使える。覚えると世界が変わる。

配列

const names2: Array<string> = ['山田', '佐藤']      // 書き方②（同じ意味）

// 複数の型が混在する配列

const point:   [number, number] = [10, 20]

// React の useState がタプルを返す
const [count, setCount] = useState<number>(0)
//     ↑string    ↑Dispatch<SetStateAction<number>>

as const — Enumより安全な定数定義

❌ enum — 問題が多い（Tree-shakingされない、型が緩いなど）
enum Status {
  Active  = 'ACTIVE',
  Inactive = 'INACTIVE',
}
// コンパイル後にオブジェクトが残る
// 数値enumは特に危険
✅ as const — 実務推奨パターン
const STATUS = {
  Active:   'ACTIVE',
  Inactive: 'INACTIVE',
} as const

// 値のUnion型を自動生成（keyof の詳細は S12 参照）
type Status = typeof STATUS[keyof typeof STATUS]
// = 'ACTIVE' | 'INACTIVE'
// as const で配列をタプルとして固定
const ALLOWED_ROLES = ['admin', 'user', 'viewer'] as const
type Role = typeof ALLOWED_ROLES[number]
// = 'admin' | 'user' | 'viewer'

satisfies 演算子（TypeScript 4.9+）

as は型を上書きするが、satisfies は「この型を満たしているか検証しつつ、推論した具体的な型を保持する」演算子。as const と組み合わせると強力。

const config = {
  host: 'localhost',
  port: 3000,
  debug: true,
} satisfies { host: string; port: number; debug: boolean }

// as で型を付けた場合: config.host は string（広い型）
// satisfies の場合: config.host は "localhost"（リテラル型を保持）

const palette = {
  red:  [255, 0, 0],
  blue: '#0000ff',  // 型が違っても OK（satisfies は各プロパティを個別検証）
} satisfies Record<string, string | number[]>

palette.red.map(String)   // ✅ red は number[] と推論される（as だと string|number[] で map できない）
palette.blue.toUpperCase() // ✅ blue は string と推論される
⚙️08 関数の型引数・戻り値・オーバーロード

TypeScriptの関数は引数と戻り値に型をつける。これにより「この関数に何を渡せるか・何が返ってくるか」をコンパイル時に保証できる。

// 基本 — 引数と戻り値に型をつける
function add(a: number, b: number): number {
  return a + b
}

// アロー関数
const add = (a: number, b: number): number => a + b

// Optional パラメータ（? をつける）
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'こんにちは'}、${name}さん`
}

// デフォルト値
function greet2(name: string, greeting: string = 'こんにちは'): string {
  return `${greeting}、${name}さん`
}

// Rest パラメータ
function sum(...numbers: number[]): number {
  return numbers.reduce((acc, n) => acc + n, 0)
}

関数型（型として定義する）

// 関数の型を変数のように定義できる
type Handler = (event: MouseEvent) => void
type Transformer<T, U> = (input: T) => U
type AsyncFetcher<T> = (id: number) => Promise<T>

// コールバックに型をつける
function withLogging<T>(fn: () => T): T {
  console.log('実行開始')
  const result = fn()
  console.log('実行完了')
  return result
}

オーバーロード — 引数パターンを複数定義する

同名の関数に「引数パターン（シグネチャ）」を複数宣言できる。実装は1つだが、呼び出し側は型に応じた適切なシグネチャとして認識される。

// 使いどころ：引数の型によって戻り値の型が変わる関数
function format(x: string): string            // シグネチャ①
function format(x: number, digits: number): string  // シグネチャ②
function format(x: any, digits?: number): string { // 実装（非公開）
  if (typeof x === 'string') return x.trim()
  return x.toFixed(digits ?? 0)
}

format('hello')     // ✅ シグネチャ① が使われる
format(3.14159, 2)  // ✅ シグネチャ② が使われる → '3.14'
format('hello', 2)  // ❌ シグネチャ① と② どちらにも合わないのでエラー
オーバーロードはライブラリ型でよく見る。自分で書く機会は多くないが、「同じ関数名なのに引数によって型が違う」という挙動の理由になる。

void vs never

void
値を返さない関数
console.logなど。returnなし or return undefinedで終わる
never
絶対に返らない関数
必ず例外をスローする・無限ループ。returnに到達しない
function log(msg: string): void {
  console.log(msg)  // return なし
}

function throwError(msg: string): never {
  throw new Error(msg)  // 必ず例外。絶対にreturnしない
}

// never は網羅チェックにも使える
function assertNever(x: never): never {
  throw new Error(`想定外の値: ${x}`)
}

🔧09 ジェネリクス型を引数として受け取る — 汎用コードの書き方

「どんな型でも使えるけど、型安全は保つ」関数・クラス・インターフェースを作る仕組み。<T> が型の仮引数（T は TypeScript の慣習的な名前で、何でもよい。複数ある場合は T, U, V や Key, Value など意味のある名前をつける）。

「引数」は値の仮引数、ジェネリクスは型の仮引数。呼び出し時に実際の型が確定する。

❌ any — 型情報が失われる
function first(arr: any[]): any {
  return arr[0]
}
const x = first([1, 2, 3])
// x は any型 → .toFixed() を呼べるか不明
✅ ジェネリクス — 型が保たれる
function first<T>(arr: T[]): T | undefined {
  return arr[0]
}
const x = first([1, 2, 3])
// x は number | undefined → 型安全
// 型パラメータは推論される（明示不要なケースが多い）
first(['a', 'b'])         // T = string と推論
first<string>(['a', 'b']) // 明示する場合（通常不要）

// 複数の型パラメータ
function pair<K, V>(key: K, value: V): [K, V] {
  return [key, value]
}
pair('name', 32)  // [string, number]

型の制約（extends）

// T は { length: number } を持つ型に限定

  return arg.length  // string[] でも string でも使える
}

// T は User のサブタイプに限定
function getEmail<T extends { email: string }>(user: T): string {

}

パターン③：Result型でエラーを安全に扱う

try/catchは「どこから例外が来るか不明」という問題がある。Result型を使うと関数の呼び出し側に「成功か失敗か」を型として強制できる。

type Result<T, E = Error> =
  | { ok: true;  data:  T }
  | { ok: false; error: E }

async function safeGetUser(id: number): Promise<Result<User>> {
  try {
    const user = await getUser(id)
    return { ok: true, data: user }
  } catch (e) {
    return { ok: false, error: e instanceof Error ? e : new Error(String(e)) }
  }
}

// 呼び出し側 — ok をチェックしないとデータにアクセスできない
const result = await safeGetUser(1)
if (result.ok) {
  console.log(result.data.name)   // User型として使える
} else {
  console.error(result.error.message)  // Error型
}

// useLocalStorage.ts
function useLocalStorage<T>(
  key: string,
  initialValue: T
): [T, (value: T) => void] {
  const [stored, setStored] = useState<T>(() => {
    try {
      const item = localStorage.getItem(key)
      return item ? (JSON.parse(item) as T) : initialValue
    } catch {
      return initialValue
    }
  })

  const setValue = useCallback((value: T) => {
    setStored(value)
    localStorage.setItem(key, JSON.stringify(value))
  }, [key])

  return [stored, setValue]  // [T, (value: T) => void] のタプル
}

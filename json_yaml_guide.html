JSON
JavaScript Object Notation

データを表現する形式。APIのレスポンスや設定ファイルなど広く使われる。型は6種類だけ。

6種類の型

文字列
"hello"
ダブルクォートで囲む
数値
42 / 3.14
整数・小数どちらもOK
真偽値
true / false
小文字のみ有効
null
null
値なしを明示する
配列
[1, 2, 3]
順番ありのリスト
オブジェクト
{"key": "val"}
キーと値のペア
全型まとめ例
// JSONはコメントが書けない。これは説明用 { "文字列": "hello", "数値": 42, "小数": 3.14, "真偽値": true, "null": null, "配列": [1, 2, 3], "オブジェクト": { "key1": "value1", "key2": "value2" } }

配列の中にオブジェクト（DBのテーブルに近いイメージ）

JSON
{ "users": [ { "name": "山田", "age": 32 }, { "name": "田中", "age": 28 } ] }

連想配列として使う（VBAのDICTIONARYと同じ感覚）

JSON
{ "fruits": { "apple": "りんご", "banana": "バナナ", "orange": "みかん" } }

詰まりやすいポイント

⚠ ケツカンマ問題

最後の要素にカンマをつけるとエラーになる。JavaScriptはOKだがJSONは厳格。

NG
"age": 32, // 最後の要素の後にカンマ
OK
"age": 32 // カンマなし
⚠ 文字列はダブルクォート必須

シングルクォートはJSONでは使えない。JavaScriptと混同しやすいので注意。

NG
'hello' // シングルクォート
OK
"hello" // ダブルクォート
⚠ 数値の落とし穴

非常に大きな数値は精度が失われることがある。巨大なIDは文字列として持つ方が安全。

JSON
// 安全な整数の上限: 9007199254740991 { "id": "12345678901234567890" } // 文字列として持つ
💡 JSONにはコメントが書けない。コメントが必要な設定ファイルにはYAMLの方が向いている。
📄
YAML
YAML Ain't Markup Language

JSONより人間が読み書きしやすい形式。設定ファイル（Docker、GitHub Actionsなど）によく使われる。インデントで構造を表現する。

基本の型・配列・オブジェクト

YAML
# 基本の型 文字列: hello 数値: 42 真偽値: true null値: null # オブジェクトの配列 users: - name: 山田 age: 32 - name: 田中 age: 28

複数行文字列 — | と > と末尾修飾子

「改行をどう扱うか」を記号で指定できる。

書き方末尾の改行結果イメージ
|1つ自動で付く"bar\nbaz\n"
|-削除する"bar\nbaz"
|+空行も全部残す"bar\nbaz\n\n"
YAML
# |- : 末尾の改行を削除（実務で一番よく使う） msg: |- 吾輩は猫である 名前はまだ無い # 結果: "吾輩は猫である\n名前はまだ無い" # >- : 改行→スペース、末尾削除 msg: >- 吾輩は猫である 名前はまだ無い # 結果: "吾輩は猫である 名前はまだ無い"
💡 実務では |- が一番よく使われる。>- は長い説明文を折り返して書きたいときに使う。

詰まりやすいポイント

⚠ インデントはスペース必須・タブNG

タブ文字を使うとエラーになる。スペース2個か4個で統一すること。

⚠ 型の自動解釈に注意

意図しない型変換が起きることがある。

YAML
flag: yes # → true として解釈される version: 1.0 # → 数値として解釈される version: "1.0" # → 文字列にしたいならクォートで囲む
💡 YAMLはコメントが書ける（# 以降）。設定ファイルに説明を残せるのが強み。
⚖️
JSON vs YAML 比較
JSON
{ "users": [ { "name": "山田", "age": 32, "active": true } ] }
YAML
users: - name: 山田 age: 32 active: true
項目JSONYAML
構造の表現波括弧・角括弧インデント
コメント書けない書ける（#）
文字列のクォートダブルクォート必須省略可
主な用途APIレスポンス・データ交換設定ファイル（Docker等）
💡 JSONの構造が理解できればYAMLも読める。形式が違うだけで表現しているデータは同じ。

エスケープ意味例
\"ダブルクォート"msg": "He said \"Hello\""
\\バックスラッシュ"path": "C:\\Users\\name"
\n改行"text": "1行目\n2行目"
\tタブ"text": "col1\tcol2"
\uXXXXUnicode文字"icon": "\u2603"
⚠ Windowsのパスは要注意
NG
{ "path": "C:\Users\name" } // \U や \n がエスケープとして解釈される
OK
{ "path": "C:\\Users\\name" } // バックスラッシュを二重にする
